# Как пользоваться этим документом

Этот файл сделан как **книга проекта** и как **навигационная карта для ИИ-агентов**.

Агенту не нужно читать всё подряд. Он должен:

1. Найти свою задачу в разделе **Маршруты чтения для агентов**.
2. Открыть нужные главы по ссылкам-якорям.
3. Прочитать только связанные разделы.
4. Выполнить bounded-задачу.
5. Обновить журнал подпроекта.

<a id="chapter-0"></a>

## 0. Назначение документа
Этот документ — главный рабочий файл проекта.

Он нужен для человека-автора проекта, Codex, OpenCode, внешнего чата, будущих AI-агентов, разработчиков, навигации по подпроектам и понимания того, к какой игре мы идём.

Документ объединяет:

```text
1. Видение игры.
2. Игровые системы.
3. Архитектуру разработки.
4. ИИ-архитектуру.
5. Редакторы и data-first подход.
6. План MVP.
7. Workflow для AI-агентов.
8. Тренировочные арены и геномы.
9. Производительность и использование ресурсов компьютера.
10. Документацию подпроектов.
```

Главная задача документа:

> **Любой агент должен прочитать этот файл и понять, к чему идёт проект, какие системы уже придуманы, какой ближайший результат нужен и как не сломать архитектуру.**

---

<a id="chapter-1"></a>

# 1. Сверхкороткая формула проекта
Игра:

> **2D тактическо-командная симуляция видом сверху, где игрок управляет не каждым солдатом, а приказами, инструктажем, планами и вмешательством через командные очки.**

Главная формула:

> **Не микроконтроль. Командование.**

Рабочая формула игрового цикла:

> **Прикажи. Наблюдай. Пойми. Исправь.**

Главная ценность:

> **Подчинённые думают, ошибаются, докладывают, боятся, держат план, меняют его при серьёзной угрозе и объясняют игроку свои решения.**

---

<a id="chapter-2"></a>

# 2. Что это за игра
Это не обычная RTS.

В обычной RTS игрок часто делает так:

```text
выбрал юнитов;
кликнул;
юниты пошли;
игрок вручную поправляет каждую мелочь.
```

В этой игре игрок должен делать другое:

```text
отдать приказ;
задать инструктаж;
посмотреть, как командир понял задачу;
увидеть план на карте;
при необходимости вручную поправить план;
потратить командные очки на вмешательство;
получать доклады;
реагировать на изменение обстановки;
решать проблемы своего уровня командования.
```

Игрок не обязан управлять каждым солдатом. Он командует через людей.

# 3. Основное ощущение от игры
Игра должна давать чувство:

```text
Я командир.
У меня есть подчинённые.
Они не роботы.
Они выполняют приказ не идеально, а как люди.
Они могут испугаться, залечь, задержаться, неправильно понять, но они не должны быть тупыми.
Если они остановились, я могу понять почему.
Если они просят поддержку, это часть боя, а не баг.
```

Ключевая эмоция:

> **доверие к подчинённым.**

Игрок должен чувствовать, что если он дал разумный приказ, подразделение постарается выполнить его разумно.

Но игрок также должен понимать:

```text
плохой командир может ошибиться;
зелёное отделение может запаниковать;
связь может задержать приказ;
доклад может устареть;
сильная угроза может сорвать план.
```

```text
ручные правки плана;
командные очки;
подчинённых, которые сами решают локальные проблемы.
```

Поэтому нужен свой **tactical simulation core**.

Важно:

> **Нужен не свой Unity/Godot, а свой боевой мозг игры.**

# 11. Главная игровая система: приказ
<a id="chapter-11-1"></a>

## 11.1. Приказ не должен быть точкой
В обычной RTS:

```text
кликнул на точку → юнит пошёл.
```

В этой игре приказ должен быть задачей:

```text
Задача: занять деревню.
Темп: осторожно.
Риск: средний.
Маршрут: желательно через лес.
Огонь: не открывать без необходимости.
Остановиться если: потери выше 25%.
При танке/ПТО: не атаковать, закрепиться, доложить.
```

<a id="chapter-11-2"></a>

## 11.2. Типы приказов
```text
двигаться;
занять позицию;
атаковать рубеж;
оборона рубежа;
подавлять сектор;
разведка боем;
обход;
закрепиться;
отойти;
держать огонь;
свободный огонь;
сопровождать танк;
следовать за танком;
занять дом;
зачистить дом;
удерживать контакт;
запросить поддержку;
блокировать цель.
```

<a id="chapter-11-3"></a>

## 11.3. Жёсткость приказа
<a id="section"></a>

### Мягкий
```text
Выполнить, если обстановка позволяет. Избегать тяжёлых потерь.
```

<a id="section-2"></a>

### Обычный
```text
Выполнить задачу, но не терять управление и не уничтожать подразделение.
```

<a id="section-3"></a>

### Жёсткий
```text
Выполнить при высоком риске. Отход только при явной невозможности.
```

<a id="section-4"></a>

### Любой ценой
```text
Задача критична. Подразделение будет больше рисковать.
```

<a id="chapter-12"></a>

# 12. Слой инструктажа приказа
<a id="chapter-12-1"></a>

## 12.1. Суть
Когда игрок отправляет юнит, например взвод, в атаку, он должен задавать не только цель, но и параметры выполнения.

Это называется:

```text
Инструктаж приказа
Order Briefing Layer
Rules of Engagement
Intent Parameters
```

Формула:

> **Приказ говорит, что сделать. Инструктаж говорит, как думать при выполнении.**

<a id="chapter-12-2"></a>

## 12.2. Что задаёт инструктаж
```text
темп;
скрытность;
строй;
уровень риска;
реакция на слабого врага;
реакция на сильного врага;
реакция на танк/ПТО;
обязательность авангарда;
порог потерь;
порог подавления;
условия отхода;
условия доклада;
правила открытия огня;
отношение к неизвестности.
```

<a id="chapter-12-3"></a>

## 12.3. Пример меню инструктажа
```text
Приказ: атаковать деревню

Темп:
[ ] Быстро
[x] Осторожно
[ ] Скрытно
[ ] Максимальный темп

Строй:
[x] Рассыпной
[ ] Колонна
[ ] Цепь
[ ] За танком
[ ] Группами по отделениям

Разведка:
[x] Обязательный авангард
[x] Проверять лесополосы
[ ] Не задерживаться на разведку
[ ] Двигаться только после подтверждения

Огонь:
[x] Открывать огонь при обнаружении
[ ] Не открывать без приказа
[ ] Подавлять подозрительные позиции
[ ] Экономить боеприпасы

Если встретили слабого противника:
[x] Подавить и продолжать
[ ] Обойти
[ ] Остановиться и доложить

Если встретили сильного противника:
[x] Закрепиться и доложить
[x] Запросить поддержку
[ ] Обойти самостоятельно
[ ] Продолжать любой ценой

Если обнаружен танк/ПТО:
[x] Не атаковать без средств
[x] Отвести людей в укрытие
[x] Доложить срочно
[ ] Попытаться обойти

Порог остановки:
Потери: 25%
Подавление: 70%
Потеря командира: пересмотреть план
Потеря связи: выполнять последний приказ 5 минут, затем закрепиться
```

<a id="chapter-12-4"></a>

## 12.4. Инструктаж как модификатор Utility AI
Инструктаж должен менять веса.

Пример:

```text
Если выбран “быстро”:
+ скорость важнее;
- скрытность менее важна;
- осторожность ниже.

Если выбран “скрытно”:
+ скрытность выше;
+ укрытия важнее;
+ открытое поле получает большой штраф;
- скорость менее важна.

Если выбран “авангард обязателен”:
план без разведки получает штраф;
план с передовым отделением получает бонус.
```

Формула:

```text
итоговая оценка =
базовая оценка плана
+ влияние приказа
+ влияние инструктажа игрока
+ влияние карты опасности
+ влияние личности командира
+ влияние опыта
+ случайное колебание
+ инерция текущего плана
```

<a id="chapter-13"></a>

# 13. Ручная правка плана
<a id="chapter-13-1"></a>

## 13.1. Суть
Игрок должен иметь возможность вручную поправить план.

Пример:

```text
Командир построил:
1. Подойти к лесополосе.
2. 1-е отделение фиксирует дом.
3. 2-е отделение обходит справа.
4. Пулемёт подавляет северный дом.
5. Атака окраины.

Игрок правит:
2-е отделение идёт не справа, а через овраг слева.
Пулемёт ставится не у леса, а у забора.
Атака начинается только после дыма.
```

<a id="chapter-13-2"></a>

## 13.2. Вес ручной правки
Правка игрока получает временно большой вес.

Пример:

```text
Правка:
2-е отделение должно идти через овраг.

Влияние:
+40 к планам, использующим овраг.
-30 к планам, игнорирующим овраг.
```

Но если обстановка резко изменилась, командир может отклониться.

Пример:

```text
Отклонение от правки:
- овраг простреливается пулемётом;
- отделение подавлено;
- выполнение приведёт к тяжёлым потерям.
```

<a id="chapter-13-3"></a>

## 13.3. Жёсткость ручной правки
```text
Рекомендация
Приоритетно
Жёсткий приказ
Любой ценой
```

Чем жёстче правка, тем сильнее вес и тем дороже вмешательство.

<a id="chapter-14"></a>

# 14. Командные очки
<a id="chapter-14-1"></a>

## 14.1. Зачем нужны
Если вмешательство игрока бесплатно, игра превратится в микроконтроль.

Командные очки нужны, чтобы:

```text
ограничить вмешательство;
сделать связь важной;
заставить выбирать приоритеты;
сохранить автономность командиров;
дать цену ручным правкам.
```

<a id="chapter-14-2"></a>

## 14.2. На что тратятся очки
```text
изменить приказ;
изменить инструктаж;
поправить маршрут;
потребовать доклад;
ускорить связь;
вручную изменить план;
передать новый приоритет;
перекинуть поддержку;
отменить атаку;
приказать продолжать любой ценой.
```

<a id="chapter-14-3"></a>

## 14.3. Цена зависит от уровня вмешательства
```text
дать общий приказ роте — 2 очка
уточнить инструктаж взводу — 3 очка
вручную править план взвода — 5 очков
отдать приказ отделению через голову взводного — 8 очков
дёргать отдельных солдат — нельзя или крайне дорого
```

<a id="chapter-14-4"></a>

## 14.4. Цена зависит от связи
```text
Есть телефон:
дёшево и быстро.

Плохое радио:
дороже, есть шанс искажения.

Только посыльный:
дорого, долго, риск потери.

Подразделение под огнём:
дороже, выше шанс неполного выполнения.
```

# 15. Слой знаний командира
<a id="chapter-15-1"></a>

## 15.1. Главный принцип
Командир не должен знать объективную истину.

Он знает:

```text
что видел сам;
что доложили подчинённые;
что сообщили соседи;
что пришло сверху;
что он предполагает;
что уже устарело;
что он вообще не знает.
```

Формула:

> **Командир решает не по реальности, а по своей картине боя.**

<a id="chapter-15-2"></a>

## 15.2. Что показывать игроку
При выборе командира игрок может включить слой:

```text
Картина боя выбранного командира.
```

Показать:

```text
точные свои;
примерные свои;
известные враги;
подозреваемые враги;
устаревшие контакты;
доклады;
источники информации;
уверенность;
свежесть сведений.
```

<a id="chapter-15-3"></a>

## 15.3. Пример
```text
Картина боя ротного:

Свои:
- 1-й взвод — точно здесь, доклад 2 минуты назад.
- 2-й взвод — примерное положение, связь нестабильна.
- ПТО — выдвигается, ETA 8 минут.

Противник:
- пулемёт — вероятная область, уверенность 70%.
- танк — район деревни, уверенность 45%.
- пехота — возможно в домах, уверенность 60%.
```

<a id="chapter-15-4"></a>

## 15.4. Структура данных

```text
  assumptions[]
  lastUpdated
}
```

Контакт:

```text

}
```

Предположение:

```text

  mayBeWrong
}
```

<a id="chapter-18"></a>

# 18. Utility AI
<a id="chapter-18-1"></a>

## 18.1. Суть
Utility AI — система оценки вариантов по баллам.

Командир получает варианты:

```text
через поле;
через лес;
по дороге;
закрепиться и запросить поддержку.
```

Каждый вариант получает оценку.

<a id="chapter-18-2"></a>

## 18.2. Факторы оценки
```text
скорость;
безопасность;
укрытия;
скрытность;
соответствие приказу;
вероятность успеха;
риск потерь;
расход ресурсов;
связь;
управление;
усталость;
наличие поддержки;
опасность;
неизвестность.
```

<a id="chapter-18-3"></a>

## 18.3. Пример оценки
```text
Обход через лес — 43

Плюсы:
+ укрытие леса: +18
+ меньше простреливается: +16
+ скрытный подход: +9
+ осторожный командир: +6
+ приказ допускает обход: +5

Минусы:
- дольше до цели: -7
- труднее держать строй: -4
- слабая связь в лесу: -3
```

<a id="chapter-18-4"></a>

## 18.4. Случайный модификатор
Командиры не должны всегда выбирать одинаково.

```text
неопытный: ±25%
обычный: ±15%
опытный: ±8%
элитный: ±5%
```

Случайность применяется только к разумным вариантам.

<a id="chapter-18-5"></a>

## 18.5. Порог разумности
Явно плохие варианты отсекаются.

Пример:

```text
Атаковать танк винтовочным взводом через поле.
Статус: запрещён / почти запрещён.
Причина: нет ПТО, открытая местность, риск критический.
```

<a id="chapter-19"></a>

# 19. Инерция плана
<a id="chapter-19-1"></a>

## 19.1. Зачем нужна
ИИ не должен дёргаться.

Плохо:

```text
увидел одного стрелка → отменил план;
услышал выстрел → развернул взвод;
доклад немного изменился → сменил маршрут.
```

Хорошо:

```text
план держится, пока событие не превышает порог пересмотра.
```

<a id="chapter-19-2"></a>

## 19.2. Бонус текущего плана
```text
Текущий план получает +5/+10 баллов.
```

Бонус снимается при важном событии:

```text
обнаружен танк;
потери 30%;
убит командир;
сосед отошёл;
появилась ПТО;
приказ изменён.
```

# 21. GOAP
<a id="chapter-21-1"></a>

## 21.1. Назначение
GOAP нужен не для всей игры, а для локальных задач.

Пример:

```text
Отделение упёрлось в простреливаемую просеку.
Нужно решить, как перейти.
```

GOAP может собрать:

```text
лечь;
найти канаву;
дождаться подавления;
перебежать;
восстановить строй.
```

<a id="chapter-21-2"></a>

## 21.2. Где использовать
```text
пересечь опасное место;
занять дом;
подавить окно;
добраться до укрытия;
выйти из-под огня;
обойти препятствие.
```

<a id="chapter-21-3"></a>

## 21.3. Где не использовать
GOAP не должен строить весь план роты.

Для роты:

```text
HTN + Utility AI.
```

Для локальной проблемы:

```text
GOAP + Behavior Tree/FSM.
```

# 22. Behavior Tree / FSM
<a id="chapter-22-1"></a>

## 22.1. Назначение
Нижний уровень исполнения.

Пример:

```text

если тяжело ранен → wounded;
если подавление выше порога → лечь;

если приказ двигаться → двигаться.
```

Состояния:

```text

moving;

suppressed;
firing;
reloading;
retreating;
wounded;
dead.
```

Behavior Tree можно добавить позже.

<a id="chapter-26"></a>

# 26. Опыт, перки и предположения
<a id="chapter-26-1"></a>

## 26.1. Опыт как умение строить гипотезы
Опытный командир лучше понимает, что может стоять за фактом.

Факт:

```text
В деревне замечен танк.
```

Зелёный:

```text
Танк в деревне.
```

Опытный:

```text
Если там танк, рядом может быть пехота прикрытия.
Могут быть пулемёты.
Деревня может быть опорным пунктом.
У дороги может быть ПТО.
```

<a id="chapter-26-2"></a>

## 26.2. Перки
```text
Опыт ПТО-засад;
Опыт штурма деревень;
Опыт лесного боя;
Опыт боя с танками;
Опыт прорыва из окружения;
Опыт обороны;
Опыт плохой связи;
Опыт зимнего/лесного боя.
```

<a id="chapter-26-3"></a>

## 26.3. Скрытая правда игры
Игра знает реальность.

Можно аккуратно использовать это, чтобы опытный командир чаще “чуял” правильную угрозу.

Правило:

> **Не “он знает”, а “он чует”.**

Пример:

```text
В лесополосе реально стоит ПТО.

Зелёный:
вероятность гипотезы 15%.

Опытный с перком ПТО-засад:
вероятность гипотезы 55%.
```

Обязательно нужны ложные тревоги и ошибки.

<a id="chapter-27"></a>

# 27. Карта опасности
<a id="chapter-27-1"></a>

## 27.1. Суть
Карта опасности показывает, что выбранный командир считает опасным.

Цвета:

```text
красное — смертельно опасно;
оранжевое — высокий риск;
жёлтое — возможная опасность;
зелёное/нейтральное — приемлемо;
серое — неизвестно.
```

<a id="chapter-27-2"></a>

## 27.2. Опасность зависит от типа юнита
Открытое поле под пулемётом:

```text
для пехоты — очень опасно;
для танка — почти безопасно, если нет ПТО;
для посыльного — очень опасно;
для орудия — плохо.
```

Дорога у лесополосы с возможной ПТО:

```text
для танка — смертельно опасно;
для пехоты — средне;
для разведчика — опасно, но можно проверить.
```

<a id="chapter-27-3"></a>

## 27.3. Причины опасности
```text
местность;
высоты;
укрытия;
видимость;
прострел;
известный враг;
предполагаемый враг;
свои силы;
наличие поддержки;
отсутствие ПТО;
качество связи;
усталость;
подавление.
```

# 29. План как граф


# 32. Тренировочные арены

## 32.1. Суть
Тренировочная арена — отдельный инструмент разработки.

Она нужна, чтобы:


```text
Пехота идёт за танком.

Цель: дойти до линии, сохранить людей и строй.
```

<a id="chapter-32-3"></a>

## 32.3. Почему headless
Миллионы прогонов нельзя считать в красивой игре.

Нужен режим:

```text
без графики;
без звука;
без UI;
без анимации;
без подробных логов;
маленькие карты;
фиксированный timestep;
батчи;
параллельные процессы.
```

<a id="chapter-34"></a>

# 34. Производительность
<a id="chapter-34-1"></a>

## 34.1. Нельзя считать всё каждый кадр
Плохо:

```text
60 раз в секунду:
каждый солдат думает;
каждый командир пересчитывает план;
вся видимость пересчитывается;
все карты влияния пересчитываются.
```

Хорошо:

```text
отрисовка — до 60 FPS;
симуляция — 10–20 тиков/сек;
решения командиров — по событиям или раз в несколько секунд;
карты влияния — частями;
видимость — кэшировать.
```

<a id="chapter-34-2"></a>

## 34.2. Использование ресурсов компьютера
```text
PixiJS/WebGL → видеокарта;
TypeScript/JS → процессор;
Web Workers → отдельные потоки браузера;
headless arena → все ядра CPU;
Rust/WebAssembly → позже, только после замеров.
```

<a id="chapter-34-3"></a>

## 34.3. Память
Плохо:

```text
каждая клетка = большой JS-объект;
каждый солдат = огромный вложенный объект;
логи бесконечные;
карта копируется часто.
```

Хорошо:

```text
высоты = Int16Array;
укрытия = Uint8Array;
опасность = Float32Array;
видимость = Uint8Array/битовые маски;
логи ограничены;
старые события архивируются.
```

# 48. Что считать успехом первого месяца
Успех — не “готовая игра”.

Успех:

```text
есть репозиторий;
есть документация;
есть каркас проекта;
есть редактор карты;
есть JSON;
есть один юнит;
есть приказ движения;
есть слой опасности;
есть первый Utility AI выбор маршрута;
есть панель объяснения.
```

Если это есть, проект реально стартовал.

<a id="chapter-49"></a>

# 49. Самая короткая итоговая формула
> Мы строим не браузерную игрушку, а локальную 2D тактическую симуляцию с веб-интерфейсом. PixiJS показывает карту, слои, юнитов и редакторы. Собственный simulation core думает: приказы, планы, знания командиров, доклады, подавление, Utility AI, HTN, GOAP, объяснения. Данные хранятся в JSON. Агенты работают через GitHub и журналы подпроектов. Первый результат — не бой, а мастерская карты и первый Tactical Utility Planner.

<a id="chapter-50"></a>

# 50. Главная установка для всех будущих агентов
> **Не делай всю игру сразу. Делай следующий маленький проверяемый слой. Если меняешь подпроект — обнови журнал. Если принял решение — запиши ADR. Если добавил систему — сделай её объяснимой. Если написал логику — отдели её от PixiJS. Если сомневаешься — сначала сохрани простоту.**
