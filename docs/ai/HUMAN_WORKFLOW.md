# Инструкция для человека

Этот документ написан для человека, который не хочет вручную работать с ветками и Pull Request.

## Главное

Вам не нужно самому делать:

```text
git checkout
git merge
git rebase
git pull
git push
```

Для этого есть Codex/OpenCode.

## Если внешний web-chat сделал Pull Request

Скажите Codex:

```text
Codex, проверь PR #N. Сам переключись на него, посмотри изменения, запусти доступные проверки и объясни мне простыми словами, что изменилось. Ничего не принимай без моего разрешения.
```

## Если Codex подготовил preview для тестирования (r-init)

Скажите Codex:

```text
Codex, что мне нужно открыть и нажать, чтобы проверить preview?
```

Codex даст вам `.bat`-файл и чеклист. Запустите файл двойным кликом и пройдите чеклист.

## GO / NO-GO

После тестирования preview скажите одно из двух:

```text
GO    — всё работает, можно мержить.
NO-GO — нужно доработать, пока не принимаем.
```

## Если результат понравился

Скажите:

```text
Codex, прими PR #N. Я разрешаю.
```

Или:

```text
Codex, можно мержить PR #N.
```

## Если результат не понравился

Скажите:

```text
Codex, не принимай PR #N. Составь замечания для автора и верни локальную папку на main.
```

## Если непонятно, что изменилось

Скажите:

```text
Codex, объясни PR #N простыми словами: что было, что стало, какие риски, что мне нужно проверить глазами.
```

## Если нужно дать задачу внешнему web-chat

Скажите:

```text
Codex, подготовь понятное задание внешнему чату. Укажи, какие файлы можно менять, какие нельзя, и что должно быть в PR.
```

## Если что-то сломалось локально

Скажите:

```text
Codex, верни локальную папку на main и объясни, что произошло.
```

## Как принимать решения

Вам нужно решать не Git-вопросы, а смысловые вопросы:

- результат полезен или нет;
- удобно или неудобно;
- надо доработать или принять;
- есть ли риск для проекта.

Техническую часть должны объяснять и выполнять помощники.
