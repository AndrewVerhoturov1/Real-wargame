# Шаблон task-pack

Этот шаблон нужен для постановки ограниченной задачи внешнему web-chat, Codex или OpenCode.

В системе есть два режима внешней работы:

- **R** — полный ручной zworker: все ссылки и правила передаются в запросе, результат — ZIP с `answer.md`.
- **Q** — короткая GitHub-aware задача: внешний исполнитель сам читает репозиторий, создаёт ветку и PR, результат — отчёт для Codex.

Если внешний исполнитель не имеет GitHub-доступа, используйте R. Если имеет GitHub-доступ и задача должна закончиться PR, можно использовать Q.

## Общий task-pack

```md
# Task

## Goal

Кратко: что нужно получить в результате.

## Context

Короткий контекст: почему задача нужна и какие ограничения важны.

## Read first

- AGENTS.md
- docs/ai/WORKFLOW_OVERVIEW.md
- docs/ai/ROLES.md

## Allowed changes

Файлы или каталоги, которые можно создавать/изменять.

## Forbidden changes

Файлы, каталоги или типы изменений, которые нельзя трогать.

## Requirements

- Требование 1
- Требование 2
- Требование 3

## Acceptance criteria

- Как понять, что задача выполнена.
- Что должно быть видно в diff/PR.
- Что должно быть объяснено в отчёте.

## Checks to run

- Команда или проверка 1
- Команда или проверка 2

Если проверок нет или они неизвестны, написать: `Needs local verification`.

## Report must include

- What changed
- Changed files
- How to check
- Checks run
- Not checked
- Risks

## Notes for human reviewer

Что человеку нужно посмотреть или решить.
```

## Правила хорошего task-pack

Хороший task-pack:

- ограничивает scope;
- говорит, что можно менять;
- говорит, что нельзя менять;
- не просит "сделать всё";
- требует честный отчёт;
- не передаёт секреты и приватные данные.

Плохой task-pack:

```text
Сделай всё красиво и наведи порядок.
```

Хороший task-pack:

```text
Обнови только docs/ai/ROLES.md и docs/ai/HUMAN_WORKFLOW.md. Не меняй код. В PR напиши, что изменилось и что не проверялось.
```

## Короткий Q-mode task-pack

Для Q-режима используйте отдельный короткий шаблон:

```text
docs/ai/TASK_PACK_Q_TEMPLATE.md
```

Минимально Q-задача должна содержать:

- Project;
- Repository;
- Subproject;
- Expected size;
- Read first;
- Goal;
- Allowed changes;
- Forbidden changes;
- Requirements;
- Output for Codex.

Q-режим не возвращает ZIP. Q-режим должен закончиться отдельной веткой, Pull Request и отчётом для Codex. Полные правила R/Q описаны в `docs/ai/ZWORKER_MODES.md`.

### R-INT task-pack

Используйте этот шаблон для постановки R-INT задачи manual zworker, когда нужно объединить несколько кандидатов (PR, ZIP, локальных патчей) в одну интеграционную ветку.

```md
# R-INT task-pack

## Goal

Объединить <кандидаты> в один интеграционный бранч.

## Candidate matrix

| Файл | PR #N | ZIP-1 | Локальный патч |
|------|-------|-------|----------------|
| path/file1.md | меняет | — | — |
| path/file2.md | — | меняет | меняет |

## Conflict matrix

| Кандидат A | Кандидат B | Пересекающиеся файлы | Риск |
|------------|------------|----------------------|------|
| PR #N | ZIP-1 | file1.md, file2.md | средний |

## Allowed changes

<что можно менять>

## Forbidden changes

<что нельзя менять>

## Requirements

1. Оценить конфликты между кандидатами.
2. Предложить порядок объединения.
3. Подготовить черновик объединённых файлов.

## Output format for zworker

- answer.md с матрицами и рекомендациями
- repo-relative файлы (черновик объединения)
```

Полные правила R-INT описаны в `docs/ai/ZWORKER_MODES.md`.
