# Post-PR consolidation

Этот документ описывает слой **после появления результатов**: PR из Q-режима, ZIP из R-режима, локальный черновик Codex/OpenCode или несколько вариантов одновременно.

Цель слоя — не потеряться в нескольких результатах и привести проект к одному понятному состоянию.

## Базовый контракт не меняется

Post-PR consolidation не отменяет главный контракт репозитория:

```text
одна задача -> одна ветка -> один Pull Request -> проверка -> решение человека
```

`main` остаётся стабильной базовой веткой. Содержательные изменения не делаются напрямую в `main`.

Если после разбора результатов нужно принять только часть PR или часть ZIP, это не значит "молча смешать всё". Безопасный путь — отдельная маленькая правка, отдельная ветка/PR или доработка существующего PR.

## Зачем нужен этот слой

Одиночный PR легко проверить по чеклисту. Но на практике может быть так:

- внешний web-chat открыл PR;
- ручной zworker вернул ZIP с похожими файлами;
- Codex сделал локальный черновик;
- OpenCode внёс маленький патч;
- часть результата полезна, а часть нет;
- несколько результатов меняют одни и те же файлы.

Codex должен не просто проверять каждый результат отдельно, а вести общую картину: что уже принято, что ещё ждёт решения, что конфликтует, что нужно показать человеку.

## Что считается входящим результатом

Входящий результат — это любой кандидат на применение:

- Q-mode PR;
- обычный PR;
- R-mode ZIP intake;
- локальная ветка Codex;
- локальный патч OpenCode;
- документальный черновик, который можно перенести в репозиторий.

## Минимальная сводка для человека

Когда результатов больше одного, Codex должен дать человеку короткую таблицу или список.

Пример:

```text
1. PR #12 — обновление документации Q-режима
   Статус: можно принять после просмотра
   Риск: низкий
   Конфликты: нет
   Ручная проверка: открыть docs/ai/ZWORKER_MODES.md и проверить, понятен ли Q-процесс

2. ZIP ZWORKER-... — альтернативный черновик
   Статус: частично полезен
   Риск: средний
   Конфликты: пересекается с PR #12 по docs/ai/CODEX_CONTROLLER_WORKFLOW.md
   Рекомендация: взять только раздел про partial acceptance, не применять весь ZIP

3. Локальный патч OpenCode
   Статус: ждёт проверки Codex
   Риск: низкий
   Ручная проверка: не нужна, правка только текстовая
```

Минимальные поля сводки:

```text
Источник
Тип: PR / ZIP / локальная ветка / патч
Статус
Кратко что меняет
Риск
Конфликты
Рекомендация
Нужна ли ручная проверка человеку
```

## Статусы входящих результатов

Использовать простые статусы:

```text
new                  новый, ещё не разобран
reviewing            Codex проверяет
needs-human-check     нужно показать человеку
accept               можно принять целиком
partial              можно принять только часть
needs-rework          вернуть на доработку
reject               отклонить
merged               принято в main
superseded            заменено другим результатом
blocked              нельзя продолжать без уточнения
included in preview   включено в r-init preview
excluded              исключено из r-init preview
needs follow-up       требует отдельной доработки после merge
```

## Как работать с несколькими PR одновременно

Codex должен:

1. Составить список PR.
2. Проверить, какие файлы меняет каждый PR.
3. Найти пересечения по файлам.
4. Найти смысловые конфликты.
5. Предложить порядок принятия.
6. Объяснить человеку, почему именно такой порядок.

Пример:

```text
Сначала принять PR #10, потому что он добавляет базовые правила.
PR #11 лучше вернуть на доработку, потому что он меняет те же файлы и частично дублирует #10.
PR #12 можно смотреть после #10, потому что он зависит от новых правил.
```

Если два PR меняют один файл, это не всегда ошибка. Но Codex должен проверить, дополняют ли они друг друга или спорят между собой.

## Как совмещать R ZIP intake и Q PR review

R ZIP и Q PR не равны по статусу.

- **Q PR** уже находится в GitHub и может быть проверен как PR.
- **R ZIP** — внешний черновик. Его нельзя считать применённым.

Порядок для R ZIP:

1. Распаковать ZIP безопасно во временное место, не в рабочий репозиторий.
2. Прочитать `answer.md` первым.
3. Посмотреть список repo-relative файлов.
4. Сравнить эти файлы с текущими файлами репозитория.
5. Решить:
   - применить всё;
   - применить часть;
   - сделать новый PR вручную/через Codex;
   - отправить на доработку;
   - отклонить.
6. Не применять ZIP напрямую в `main` без отдельного решения.

Если ZIP и PR меняют одни и те же файлы, Codex должен объяснить человеку конфликт и предложить один путь:

```text
Вариант A: принять PR, а ZIP считать устаревшим.
Вариант B: взять из ZIP только один раздел и добавить его отдельным PR.
Вариант C: попросить внешнего исполнителя обновить PR с учётом ZIP.
```

## Partial acceptance

Partial acceptance — это когда результат полезен не целиком.

Примеры:

- из PR полезен один документ, но не весь набор;
- из ZIP полезен один раздел;
- идея хорошая, но реализация слишком широкая;
- часть файлов конфликтует с уже принятым результатом.

Правило:

```text
Если принято только часть результата, нельзя мержить весь PR как есть.
```

Codex должен:

1. Назвать, что принимается.
2. Назвать, что не принимается.
3. Объяснить почему.
4. Предложить безопасный способ переноса:
   - попросить автора сузить PR;
   - создать отдельный маленький PR;
   - перенести один файл/раздел вручную;
   - попросить OpenCode сделать ограниченный патч.

Пример вывода человеку:

```text
Я предлагаю частично принять результат: взять раздел "ручная проверка человеком" из ZIP, но не применять остальное. Остальное дублирует уже принятый PR #14. Безопасный путь — создать новый маленький PR только с этим разделом.
```

## Ручная проверка человеком

Codex должен давать инструкции ручной проверки там, где человеку реально нужно посмотреть результат.

Это особенно важно, если результат влияет на:

- поведение программы;
- интерфейс;
- внешний вид;
- удобство;
- понятность текста;
- важное организационное решение;
- то, что нельзя уверенно оценить только по diff.

Инструкция должна быть короткой и практичной:

```text
Что открыть:
- ...

Что сделать:
- ...

Ожидаемый результат:
- ...

Если не так:
- пришли скриншот / текст ошибки / короткое описание
```

Пример для документации:

```text
Открой docs/ai/ZWORKER_MODES.md.
Проверь, понятно ли различие R и Q без прошлого контекста.
Если непонятно, напиши какие слова сбивают.
```

Пример для работающего проекта:

```text
Запусти проект обычным способом, если доступно.
Открой страницу, которую менял PR.
Проверь: кнопка видна, действие выполняется, ошибок на экране нет.
Если что-то не так, пришли скриншот или текст ошибки.
```

Если ручная проверка не нужна, Codex должен прямо написать:

```text
Ручная проверка человеком не обязательна: изменение только в служебной документации, достаточно проверить diff и пути файлов.
```

## Preview-схема

Проект использует двухпапочную схему: `Real-wargame` (main) и `Real-wargame-preview` (real-wargame-preview). Результаты задач сначала попадают в preview-ветку, тестируются пользователем в preview-папке, и только после GO переносятся в `main`. Подробно: `docs/workflow/LOCAL_TWO_FOLDER_WORKFLOW.md`.

## После merge или применения результата

После того как человек разрешил принять результат, Codex должен привести проект к единому состоянию.

Минимальный порядок:

1. Зафиксировать, какой результат принят: PR/ZIP/локальный патч.
2. Если это PR — смержить только после явного разрешения человека.
3. Если это ZIP — применить только согласованные файлы/части через отдельную контролируемую правку, а не "как попало".
4. Вернуть локальную папку на `main` или согласованную базовую ветку.
5. Обновить локальную папку из GitHub, если есть доступ.
6. Проверить, что принятый результат действительно попал в базу.
7. Запустить доступные проверки или написать `Needs local verification`.
8. Дать человеку ручную инструкцию проверки, если она уместна.
9. Обновить сводку статусов: принято, отклонено, ждёт доработки, заменено.

## Итоговый отчёт Codex после consolidation

После разбора нескольких результатов Codex должен дать человеку такой отчёт:

```text
Итог по входящим результатам

Принято:
- PR #12: ...

Частично принято:
- ZIP ZWORKER-...: взят только раздел ...

На доработку:
- PR #13: причина ...

Отклонено/заменено:
- локальный патч ...

Текущее состояние проекта:
- main обновлён / main не обновлялся
- локальная папка синхронизирована / нужна синхронизация
- проверки: ...

Что человеку проверить вручную:
- ...

Следующий безопасный шаг:
- ...
```

## Чего не делать

Codex не должен:

- мержить несколько PR подряд без отдельного понятного вывода;
- применять ZIP прямо в `main`;
- смешивать частично принятые куски без объяснения;
- скрывать конфликты между результатами;
- говорить "всё принято", если часть результата осталась вне проекта;
- забывать вернуть локальную папку к согласованной базе;
- требовать от человека самому разруливать ветки;
- забывать дать ручную инструкцию проверки, когда результат нужно смотреть глазами.
