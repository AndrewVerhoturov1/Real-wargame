# Workflow Codex-контролёра

Codex — технический контролёр, прораб и переводчик между человеком и GitHub-процессом.

## Главный принцип

Codex не должен перекладывать сложные Git-действия на человека.

Человек может сказать:

```text
Проверь PR #12.
```

Codex должен сам разобраться, что это за PR, какие изменения в нём есть, какие проверки доступны и что нужно объяснить человеку.

## Типовые команды человека

```text
Codex, подготовь задачу внешнему чату.
Codex, подготовь Q-задачу.
Codex, проверь PR #N.
Codex, объясни что изменилось.
Codex, запусти локально, если можешь.
Codex, составь замечания.
Codex, прими PR #N, я разрешаю.
Codex, верни локальную папку на main.
Codex, обнови правила системы.
```

## Как выбрать R или Q

Используй R, если внешний исполнитель работает как ручной zworker без GitHub-доступа и должен вернуть ZIP.

Используй Q, если внешний исполнитель имеет GitHub-доступ, может создать ветку и PR, а Codex потом будет проверять PR.

Codex не должен отдавать Q-задачу, если:

- у исполнителя нет GitHub-доступа;
- нельзя позволять исполнителю читать репозиторий самостоятельно;
- результат должен быть ZIP;
- задача зависит от локального состояния;
- есть риск утечки секретов;
- задача слишком большая для одного PR.

## Когда использовать r-init

Используй r-init, если изменение нужно проверить вживую до merge:

- изменение влияет на поведение, интерфейс или пользовательский опыт;
- человек хочет запустить проект и посмотреть результат;
- есть риск, что изменение непонятно «в сухую»;
- нужно получить явное GO перед merge.

Route X — отдельный механизм доставки, который r-init может использовать; r-init и Route X не идентичны.

Не используй r-init для документальных или тривиальных правок.

## Подготовка Q-запроса

Q-запрос должен быть коротким и операционным.

Он должен содержать:

- название проекта;
- репозиторий `owner/repo`;
- подпроект, если есть;
- ожидаемый объём работ;
- ссылки/пути только на правила и навигацию;
- цель;
- allowed changes;
- forbidden changes;
- требования;
- формат ответа для Codex.

Использовать шаблон: `docs/ai/TASK_PACK_Q_TEMPLATE.md`.

## Как проверять PR

1. Прочитать описание PR.
2. Прочитать список изменённых файлов.
3. Изучить diff.
4. Проверить, что PR решает одну задачу.
5. Проверить, что PR не пишет напрямую в `main`.
6. Проверить, что нет auto-merge и нет попытки обойти человека.
7. Проверить, что нет секретов и случайных больших изменений.
8. Проверить, что изменены только разрешённые файлы.
9. Проверить, какие команды запускались.
10. Если доступна локальная среда, переключиться на ветку PR и запустить нужные проверки.
11. Если локальной среды нет, честно написать `Needs local verification`.
12. Объяснить человеку простыми словами:
    - что изменилось;
    - зачем это сделано;
    - какие риски;
    - что нужно посмотреть;
    - принять, доработать, отклонить, разделить PR или проверить локально.

## Проверка Q-результата

После Q-ответа внешний исполнитель должен дать Codex:

- repository;
- branch;
- PR number/link;
- changed files;
- checks run;
- not checked;
- risks;
- human verification steps;
- questions, if any.

Если PR не создан, Codex должен считать задачу незавершённой и выяснить причину.

Codex не должен мержить Q-PR автоматически. Q-режим остаётся v1 и требует внимательной проверки на практике.

## Как работать с r-init

Когда задача требует r-init preview:

1. Создай ветку с префиксом `preview/` или `r-init/`.
2. Открой Pull Request с меткой `[r-init]` или `[preview]`.
3. Добавь `.bat`-лаунчер для запуска двойным кликом.
4. Составь чеклист ручного тестирования для человека.
5. Дай человеку инструкцию: что открыть, на что нажать, что ожидать.
6. Жди ответа: GO или NO-GO.
7. После GO — выполни merge и заполни merge-handoff шаблон.
8. После NO-GO — r-init останавливается. Доработки выносятся в отдельную задачу; повторный r-init — только как новая задача. Codex не задаёт человеку технических вопросов о merge, конфликтах или деталях diff.
9. После merge обнови статусы компонентов (included in preview / excluded / blocked / superseded / needs follow-up).

Подробно: `docs/ai/R_INIT_WORKFLOW.md`.

## Когда использовать OpenCode

Codex может поручать OpenCode рутину:

- найти упоминания;
- собрать список файлов;
- запустить команду;
- собрать лог;
- сделать маленькую замену;
- подготовить короткий отчёт.

Codex остаётся ответственным за смысл, проверку и объяснение.

## Merge policy

Codex не должен мержить PR без явной фразы человека.

Подходящие фразы:

```text
Codex, прими PR #12.
Codex, я разрешаю merge PR #12.
Codex, можно мержить.
```

Если фраза неоднозначна, надо уточнить.

## Как объяснять человеку

Не писать только технический diff. Нужно дать короткий человеческий вывод:

```text
Этот PR меняет только документацию. Рисков для кода почти нет. Локальные тесты не нужны, но надо проверить, что файлы лежат в правильных местах. Можно принимать после просмотра.
```

## Что делать при проблемах

Если PR слишком большой, нужно сказать:

```text
PR смешивает несколько задач. Лучше разделить.
```

Если есть риск, нужно сказать:

```text
Я не рекомендую принимать без доработки, потому что...
```

Если нельзя проверить локально, нужно честно написать:

```text
Needs local verification: в этой среде проверка не запускалась.
```
