# Codex handoff — первый видимый шаг Real wargame

Дата: 2026-07-06  
Подпроект: `real-wargame-start`  
Режим: Q / GitHub-aware / без локальных проверок  
Ветка контекста: `codex/patch-pack-subproject-context`

## Что решили

После обсуждения первый видимый шаг проекта лучше сместить с чистого редактора карты на маленький интерактивный тактический прототип.

Рабочее название:

> **Tactical Board Prototype v0.1**

Это не отменяет `Tactical Map Workshop v0.1`, но меняет практический старт: сначала нужен живой экран, где уже чувствуется будущая игра.

## Почему так

Редактор карты полезен для производства, но сам по себе он не показывает core loop игры. Пользователю сейчас важнее получить видимый результат:

```text
есть карта;
есть зум;
есть пешки бойцов/отделений;
пешки выделяются;
правой кнопкой им отдаётся приказ;
они движутся к точке;
видна линия приказа.
```

Это лучше проверяет базовое ощущение проекта:

```text
я командир;
у меня есть подразделения;
я выбираю подразделение;
я отдаю приказ;
оно выполняет приказ.
```

## Минимальный scope Tactical Board Prototype v0.1

Сделать один маленький bounded-пакет:

1. Vite + TypeScript + PixiJS.
2. Один экран тактической карты.
3. Простая карта-сетка 20×20, клетка 32 px.
4. Камера:
   - zoom колесом мыши;
   - желательно pan/drag по карте, если не раздувает задачу.
5. 2–5 тестовых пешек бойцов/отделений.
6. Клик по пешке выделяет её.
7. Выделенная пешка подсвечивается рамкой/кольцом.
8. Правый клик по карте создаёт приказ движения для выбранной пешки.
9. От выбранной пешки к цели рисуется линия приказа.
10. Пешка движется к точке простым прямолинейным движением.
11. Debug overlay:
    - координаты мыши в клетках;
    - id выбранной пешки;
    - текущая цель приказа.

## Архитектурное правило

Даже в минимальном прототипе не надо хранить игровую логику внутри Pixi-объектов.

Нужно сохранить направление из GDD:

```text
PixiJS = экран / rendering / input.
Core = состояние карты, юнитов, приказов и simulation tick.
```

Минимальная желательная структура:

```text
src/
  core/
    map/
      MapModel.ts
    units/
      UnitModel.ts
    orders/
      MoveOrder.ts
    simulation/
      SimulationState.ts
      SimulationTick.ts

  rendering/
    PixiApp.ts
    PixiMapRenderer.ts
    PixiUnitRenderer.ts
    PixiOrderRenderer.ts

  input/
    CameraController.ts
    SelectionController.ts
    OrderInputController.ts

  data/
    maps/test_map.json
    units/test_units.json
```

Если для первого пакета часть файлов будет объединена, это допустимо, но направление должно быть таким: core не импортирует PixiJS.

## Предлагаемый порядок первых шагов

1. **Tactical Board Prototype v0.1** — карта, зум, пешки, выделение, приказ движения.
2. **Map Data / Import** — карта и юниты грузятся из JSON.
3. **Simple Map Editor** — terrain-кисти, высоты, export/import JSON.
4. **First Tactical Meaning** — terrain начинает влиять на скорость/стоимость движения.
5. **Danger Map** — пулемётная точка, сектор огня, красная зона опасности.
6. **First AI Choice** — простой выбор маршрута по баллам и объяснение причин.

## Просьба к Codex

Codex должен уточнить и явно перечислить:

1. Что именно внешний чат **точно не сможет сделать** в этом режиме.
2. Какие действия требуют локального запуска, локальной проверки, визуального просмотра или доступа к окружению пользователя.
3. Какие части задачи Codex должен оставить за собой как контролёр/проверяющий.
4. Какие части можно и нужно передать внешнему чату как исполнителю через GitHub.

Главное правило для следующей работы:

> Всё, что внешний чат может выполнить через GitHub, редактирование файлов и текстовые инструкции, должен выполнять внешний чат. Codex не должен тратить дорогую работу на то, что можно вынести во внешний чат.

Codex должен оставить себе только:

```text
контроль архитектурных границ;
проверку соответствия GDD;
локальные проверки, если они реально нужны и доступны;
визуальную проверку, если она невозможна для внешнего чата;
финальное ревью результата;
решения, которые требуют доступа к локальной среде пользователя.
```
