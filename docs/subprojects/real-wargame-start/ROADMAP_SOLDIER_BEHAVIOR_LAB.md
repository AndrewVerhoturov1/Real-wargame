# Roadmap — Soldier Behavior Lab

Дата: 2026-07-07  
Подпроект: `real-wargame-start`  
Статус: рабочий план поэтапного развития игры после первого tactical board prototype

## Смысл плана

Главная ценность Real wargame — не обычная RTS, а низовой интеллект солдат и командование через приказы, наблюдение, понимание причин поведения и ограниченное вмешательство.

Поэтому развитие проекта нужно строить не только вокруг карты или редактора карты, а вокруг связки:

```text
поведение солдата;
объяснение поведения;
редактор параметров поведения;
отладочные панели;
слои знаний/опасности/видимости.
```

Ключевая формула:

> Сначала делаем удобный RTS-стенд. Потом сразу делаем лабораторию поведения солдата: инспектор, параметры, состояния, события, зрение, знания, scoring. Редактор поведения появляется почти сразу, но маленький, конкретный и растущий вместе с поведением.

## Что значит “видимый результат этапа”

После каждого этапа должен быть результат, который можно показать в короткой записи экрана или просто руками продемонстрировать в браузере.

Это не обязательно публичное видео. Смысл в том, чтобы пользователь видел прогресс и не перегорал:

```text
открыл прототип;
выделил солдата;
изменил параметр;
солдат повёл себя иначе;
панель объяснила почему.
```

Плохая задача:

```text
подготовить архитектуру под будущую систему поведения.
```

Хорошая задача:

```text
солдат увидел опасность, залёг, а панель показала: danger 80 > threshold 60.
```

## Общий принцип разработки

Каждая новая возможность поведения должна идти комплектом:

```text
1. само поведение;
2. debug-панель “что происходит и почему”;
3. маленький редактор параметров;
4. тестовая ситуация на карте.
```

Не делать поведение вслепую. Если солдат что-то делает, пользователь должен видеть:

```text
что он делает;
почему он это делает;
какие параметры повлияли;
как изменить эти параметры;
что изменилось после правки.
```

## Блок 0 — Tactical Board Prototype

Цель: получить удобный тактический стенд, а не полноценную RTS.

Минимум:

```text
карта;
зум;
pan камеры;
пешки/солдаты;
выделение;
правый клик для приказа движения;
линии приказов;
HUD/debug panel.
```

Не делать на этом этапе:

```text
экономику;
строительство;
ресурсы;
полноценный бой;
сложный AI;
кампанию.
```

Назначение этапа:

> У нас есть удобный стол, на котором можно наблюдать и отлаживать поведение солдат.

## Блок 1 — Tactical Board Polish

Цель: сделать стенд достаточно приятным, чтобы на нём удобно тестировать интеллект.

Задачи:

```text
улучшить читаемость пешек;
сделать удобный zoom к курсору;
сделать удобный pan;
сделать понятную подсветку выбранного юнита;
добавить журнал приказов;
добавить help-панель;
добавить reset demo;
сделать линии приказов и цели хорошо читаемыми.
```

Критерий:

```text
пользователь может открыть прототип, выделить юнита, дать приказ, наблюдать движение и не бороться с управлением.
```

## Блок 2 — Soldier Behavior Lab

Это большой блок, который объединяет раннее поведение солдат и инструменты понимания/редактирования.

Важно: редактор поведения не откладывается на конец. Он появляется рано, но сначала как простой отладочный UI.

### 2.1. Soldier Inspector v0.1

Цель: клик по солдату показывает, что он делает и почему.

Панель выбранного солдата:

```text
id;
name;
state;
current_order;
current_action;
reason;
position;
target;
last_event.
```

Пример:

```text
Soldier: Ivanov
State: moving
Order: move_to_point
Action: walking
Reason: executing direct move order
Position: 5, 8
Target: 12, 10
```

Критерий:

```text
кликаю на солдата — справа вижу его состояние, приказ, действие и причину.
```

### 2.2. Behavior Parameters v0.1

Цель: первые редактируемые параметры поведения.

Параметры:

```text
obedience;
caution;
aggression;
fear;
move_speed;
danger_stop_threshold.
```

Интерфейс:

```text
простая HTML-панель;
input number / sliders;
Apply;
Reset.
```

Критерий:

```text
меняю move_speed — солдат реально идёт быстрее/медленнее;
меняю threshold — в debug-панели видно новое значение.
```

### 2.3. Behavior Profiles v0.1

Цель: разные типы солдат ведут себя по-разному.

Профили:

```text
green;
regular;
veteran;
cautious;
reckless.
```

Пример данных:

```text
green:
  obedience: 0.7
  caution: 1.4
  panic_threshold: 45

veteran:
  obedience: 1.1
  caution: 1.0
  panic_threshold: 75
```

Критерий:

```text
в одной и той же ситуации green солдат останавливается/пугается раньше, veteran держится дольше.
```

### 2.4. First State Machine v0.1

Цель: солдат имеет понятные состояния.

Состояния:

```text
idle;
moving;
observing;
stopped;
taking_cover;
confused.
```

Примеры переходов:

```text
нет приказа -> idle;
есть move order -> moving;
дошёл до точки -> observing;
danger > threshold -> taking_cover.
```

Inspector должен показывать:

```text
state;
previous_state;
state_changed_because.
```

Критерий:

```text
солдат шёл, попал в опасную ситуацию, сменил состояние, а панель объяснила почему.
```

### 2.5. Rule Editor v0.1

Цель: первый простой редактор правил поведения.

Это не большой визуальный node-editor. Это таблица:

```text
Condition                         Action
danger > danger_stop_threshold    take_cover
enemy_visible == true             stop_and_observe
order == move && danger low       continue_move
reached_target == true            observe
```

Ограничения v0.1:

```text
условие выбирается из списка;
оператор выбирается из списка;
значение вводится числом/boolean;
действие выбирается из списка;
никакого произвольного программирования пользователем.
```

Критерий:

```text
меняю threshold с 70 на 40 — солдат реагирует раньше;
меняю action с take_cover на stop_and_observe — реакция меняется.
```

### 2.6. Event Debugger v0.1

Цель: тестировать интеллект без полноценного боя.

Кнопки событий:

```text
Simulate enemy spotted;
Simulate shot nearby;
Simulate order received;
Simulate lost contact;
Simulate messenger arrived;
Clear events.
```

Inspector показывает:

```text
last_event;
reaction;
reason;
changed_parameters.
```

Пример:

```text
last_event: shot_nearby
reaction: take_cover
reason: shot_nearby increased danger to 85
```

Критерий:

```text
нажимаю shot nearby — солдат меняет состояние, панель объясняет реакцию.
```

### 2.7. Perception v0.1

Цель: солдат видит не всё, а только в пределах органов чувств.

Добавить:

```text
направление взгляда;
угол обзора;
дальность обзора;
сектор обзора на карте;
visible_contacts.
```

Inspector показывает:

```text
view_angle;
view_range;
looking_direction;
visible_contacts.
```

Критерий:

```text
поворачиваю солдата — сектор обзора меняется;
враг попал в сектор — появился visible contact.
```

### 2.8. Knowledge v0.1

Цель: отделить объективную карту от знания конкретного солдата.

Солдат хранит:

```text
known_contacts;
last_seen_position;
last_seen_time;
confidence;
heard_events;
suspected_danger_zones.
```

Режимы просмотра:

```text
Objective view;
Soldier knowledge view;
Player knowledge view.
```

Критерий:

```text
на объективной карте враг есть;
в знаниях солдата врага нет;
солдат увидел врага;
в его знаниях появился контакт.
```

### 2.9. Scoring v0.1

Цель: солдат выбирает действие по баллам, а не по жёсткому одному if.

Действия-кандидаты:

```text
continue_move;
stop_and_observe;
take_cover;
move_to_cover;
report_contact;
retreat_short.
```

Decision Debugger показывает:

```text
Action: continue_move
+ order obedience: 50
- danger: 80
= score: -30

Action: take_cover
+ survival: 70
- order delay: 20
= score: 50

Selected: take_cover
```

Редактор весов:

```text
order_weight;
danger_weight;
cover_weight;
contact_weight;
panic_weight.
```

Критерий:

```text
солдат под угрозой;
таблица показывает 2-3 варианта;
выбран вариант с лучшим score;
меняю danger_weight — выбранное действие может измениться.
```

## Блок 3 — Карта и среда для интеллекта

Карта нужна не как картинка, а как источник условий для поведения.

### 3.1. Map Coordinates v0.1

Цель: карта должна быть полностью понятна движку.

Добавить/уточнить:

```text
world coordinates;
grid coordinates;
cell coordinates;
конвертация screen <-> world <-> grid;
стабильный формат JSON карты.
```

Критерий:

```text
по клику на карту движок точно знает клетку, world position и объект под курсором.
```

### 3.2. Map Editor v0.1

Цель: простой, но функциональный редактор карты.

Редактировать:

```text
terrain;
height;
cover;
movement_cost;
vision_blocking / transparency;
spawn points;
test markers.
```

Функции:

```text
brush tools;
erase/reset;
export JSON;
import JSON;
load test maps.
```

Критерий:

```text
пользователь рисует лес/дорогу/болото, сохраняет JSON, загружает карту и солдаты используют эти данные.
```

## Блок 4 — Слои

Слои — ключевой способ понять, почему интеллект действует так, а не иначе.

### 4.1. Terrain/Movement Layer

Показывает:

```text
terrain;
movement cost;
проходимость;
скорость.
```

### 4.2. Visibility Layer

Показывает:

```text
что видно объективно;
что видит выбранный солдат;
что блокирует обзор;
сектор обзора;
line of sight.
```

### 4.3. Danger Layer

Показывает:

```text
опасные клетки;
источники опасности;
уровень опасности;
почему клетка опасна.
```

### 4.4. Knowledge Layer

Показывает:

```text
что знает игрок;
что знает выбранный солдат;
известные контакты;
подозреваемые зоны;
устаревшую информацию.
```

Критерий блока:

```text
можно переключить objective view, soldier knowledge view, danger view и понять, почему солдат выбирает действие.
```

## Блок 5 — План как объект

Цель: перейти от одного действия к плану из шагов.

План должен хранить:

```text
plan_id;
goal;
steps;
risk;
expected_time;
score;
reasons;
abort_conditions;
replan_conditions.
```

Пример:

```text
Plan: reach_tree_line
Steps:
1. move to road bend
2. cross field quickly
3. stop at forest edge
4. observe target area

Abort if:
- suppression > 60
- enemy spotted within 4 cells
- leader wounded
```

Критерий:

```text
солдат получил задачу;
игра показывает его план;
появился новый фактор;
план отменился или пересчитался;
панель объяснила причину.
```

## Три инструмента редактора интеллекта

Редактор ИИ лучше разделить на три части.

### 1. Inspector

Показывает состояние:

```text
что солдат делает;
что он знает;
что он видит;
какой приказ выполняет;
какое состояние;
какое последнее событие.
```

### 2. Parameter Editor

Позволяет менять числа и профили:

```text
порог страха;
осторожность;
послушание;
скорость;
дальность обзора;
вес опасности.
```

### 3. Decision Debugger

Показывает причины решения:

```text
какие варианты были;
какие баллы у каждого;
какой вариант победил;
какие факторы повлияли.
```

## Важные анти-цели

Не делать сразу:

```text
универсальный редактор всего;
сложный визуальный node-editor;
полный GOAP/HTN;
полноценную симуляцию боя;
реалистичную баллистику;
кампанию;
мультиплеер;
красивую графику вместо отладки поведения.
```

Пока цель другая:

```text
солдат имеет мозг;
пользователь видит этот мозг;
пользователь может менять параметры;
поведение меняется;
причины поведения понятны.
```
