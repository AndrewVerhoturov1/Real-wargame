# RTS-заготовка — ключевые решения

Этот файл фиксирует устойчивые решения по текущей RTS-заготовке. Это не финальная игра, а рабочая основа для дальнейшей разработки поведения солдат.

## Статус

Этап RTS-заготовки в целом готов. Его не надо постоянно перепроектировать с нуля. Дальше карта, интерфейс, редактор, видимость, укрытия и производительность правятся по ходу работы над “мозгами солдат”.

## Главная цель

Проект сейчас — **Soldier Behavior Lab**:

- карта должна давать солдату понятную среду;
- юнит должен получать знания об укрытиях, опасности, рельефе и видимости;
- редактор должен позволять быстро собирать тестовые ситуации;
- интерфейс должен помогать человеку видеть, что юнит знает и почему он принимает решение.

Это не стратегический ИИ командира. Цель — поведение отдельных солдат/малых групп в тактической среде.

## Масштаб и карта

Текущая карта:

```text
width: 64
height: 40
cellSize: 24
metersPerCell: 10
```

Решение: считать `1 клетка = 10 м`. В интерфейсе и инспекторе показывать расстояния в метрах, а не в клетках.

## Высоты

Высоты карты:

```text
-2, -1, 0, +1, +2, +3, +4
```

Решения:

- не показывать на карте постоянную россыпь `+1` / `-1`;
- визуально показывать высоты как физическую карту: цветные области, кривые границы, тонкие линии уровней;
- цифры высот оставить только как отладочный переключатель;
- линии высот — визуальная подсказка, а не вертикальные стены;
- реальный расчёт видимости должен использовать сглаженный рельеф.

## Реальный рельеф

Есть отдельный верхний переключатель:

```text
Реальный рельеф: вкл/выкл
```

Этот слой показывает фактическую сглаженную высоту, которую использует расчёт видимости. Он нужен для отладки и объяснения, почему линия видимости проходит или прерывается.

Производительное решение:

- слой `Реальный рельеф` отделён от динамической подсветки мыши;
- слой кэшируется;
- движение мыши, выделение клетки и Alt-линия не должны пересобирать весь слой рельефа;
- не возвращаться к `antialias: false`, потому что это портит внешний вид.

## Лес

Лес — отдельный слой карты:

```text
0 — нет леса
1 — редкий лес
2 — густой лес
```

Решения:

- лес можно рисовать кистью в редакторе;
- лес влияет на знания юнита и линию видимости;
- ближний лес в знаниях юнита показывать подробнее;
- дальний лес показывать компактными ориентирами, чтобы не засыпать карту сотнями меток.

## Объекты и физическая высота

Предметы карты имеют размеры в клетках и физическую высоту для видимости:

```text
widthCells
heightCells
losHeightMeters
```

Примеры высоты по умолчанию:

```text
дерево: 6 м
дом: 5 м
пост / бочки: 1.35 м
ящики / бочки: 1.25 м
камень: 1.2 м
забор: 1.2 м
укрытие: 1.1 м
брёвна: 0.8 м
канава: 0.2 м
```

Решение: объект не должен быть бесконечной стеной. Линия видимости учитывает:

1. высоту земли под наблюдателем;
2. высоту земли под целью;
3. сглаженную высоту рельефа между ними;
4. физическую высоту объекта;
5. позу/высоту глаз юнита.

Если солдат стоит выше, низкие бочки внизу не должны автоматически закрывать весь обзор. Дом или дерево должны закрывать сильнее.

## Укрытия и знания юнита

Во вкладке `Слои` выбранного юнита показывать:

- ближние укрытия;
- дальние укрытия для плана;
- опасность.

Решения:

- ближние укрытия нужны для текущего использования;
- дальние укрытия нужны для планирования маршрута;
- укрытие должно иметь смысл по расстоянию и/или видимости;
- все расстояния показывать в метрах;
- ближние укрытия на карте — квадратами;
- дальние ориентиры/укрытия — кругами;
- опасность — красными метками.

## Alt-линия видимости

Режим проверки:

```text
выбрать юнита -> зажать Alt -> вести мышь по карте
```

Должно быть видно:

- зелёная линия — видимая часть;
- красная линия — участок за преградой;
- красный крест — место прерывания;
- HTML-плашка с расстоянием в метрах;
- причина преграды: рельеф, лес, дом, камень, бочки/ящики и т.п.

Текст линии видимости должен быть HTML-плашкой, не пиксельным текстом Pixi.

## Игровой интерфейс

Игровой режим не должен быть длинной правой простынёй.

Текущая структура:

```text
верхняя панель: режим, сетка, реальный рельеф, цифры высоты, язык
правая панель: вкладки Юнит / Слои / Поведение / Сенсоры
нижняя панель: клетка под мышью + карточка выбранного юнита
карта: основное рабочее пространство
```

Решение: карточка юнита должна быть стабильного размера и не прыгать от количества данных.

## Редактор

Редактор — отдельный режим, а не смесь с игровым интерфейсом.

Текущая структура:

```text
Инструмент
Карта
Создать
Выбранное
Сцена
```

Редактор умеет:

- создавать предметы;
- создавать юниты;
- создавать зоны;
- двигать/масштабировать/вращать предметы;
- двигать/масштабировать зоны;
- рисовать высоты;
- рисовать лес;
- очищать сцену;
- скачивать JSON сцены;
- загружать JSON сцены;
- скачивать отчёт производительности.

## Производительность и вид

Решения:

- `antialias: true` оставить включённым;
- `antialias: false` не использовать как постоянное решение, потому что картинка становится грубой;
- view cones выключены по умолчанию;
- цифры высот выключены по умолчанию;
- земля рисуется пачками по типам/слоям, а не тысячами отдельных объектов;
- динамическая подсветка мыши не должна пересобирать зоны и рельеф;
- реальный рельеф кэшируется отдельным слоем.

## Пользовательская проверка

Главный запуск для человека:

```text
scripts/windows/run-preview.bat
```

Не просить человека запускать Git, терминал, checkout, merge, rebase или другие технические команды.

Отчёт человеку давать в формате:

```text
что сделано
где сделано
что проверить
GO / NO-GO
что не проверялось
```

## Что не является частью готового этапа

Пока не считать готовыми:

- полноценный бой;
- баллистику;
- pathfinding;
- мораль;
- связь между солдатами;
- передачу знаний между юнитами;
- сложный ИИ выбора маршрута;
- стратегический ИИ командира;
- финальный художественный вид карты.

Эти части должны добавляться отдельными этапами поверх текущей RTS-заготовки.
