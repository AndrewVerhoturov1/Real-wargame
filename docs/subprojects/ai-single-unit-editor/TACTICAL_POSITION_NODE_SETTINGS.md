# Настройки CreateTacticalPositionCandidates

## Назначение

`CreateTacticalPositionCandidates` создаёт наблюдательные, оборонительные и огневые позиции. Нода управляет только субъективным запросом конкретного бойца: что считать хорошей позицией, какие позиции отбрасывать и сколько вычислений разрешить.

Изменение этих полей:

- создаёт новую идентичность субъективного запроса;
- отменяет переиспользование старого готового ответа;
- не меняет идентичность постоянной тактической основы;
- не перестраивает карту статических кандидатов;
- проходит через граф, хозяина симуляции, службу поиска и рабочий поток одним типизированным объектом.

Единственный источник значений по умолчанию и нормализации находится в `TacticalPositionNodeSettingsRuntime.ts`. Интерфейс, раннер и поисковый движок используют его напрямую.

## Группы в редакторе

Панель свойств разделена на сворачиваемые секции:

1. **Основные** — тип позиции, задача движения, цель, радиус и число результатов.
2. **Веса оценки** — состав итоговой мягкой оценки.
3. **Задача движения** — сила сближения, отхода и продолжения приказа.
4. **Ограничения** — условия жёсткого отбрасывания.
5. **Поза** — допустимые позы и цена перехода.
6. **Производительность — расширенные** — жёсткие пределы вычислений.

У каждого поля есть диапазон, шаг, единица, подсказка и отдельный сброс. У каждой группы есть общий сброс. Расширенные группы изначально свёрнуты.

## Основные параметры

| Поле | По умолчанию | Диапазон | Что меняется |
|---|---:|---:|---|
| `kind` | `defense` | observation / defense / firing | Вид искомой позиции и начальные веса типа. |
| `objective` | `balanced` | balanced / advance / withdraw / continue | Направление, которое предпочитает итоговое ранжирование. |
| `queryKey` | `tactical_position_query` | непустая строка | Связь нод создания, фильтрации и выбора. |
| `targetMode` | `automatic` | automatic / order_point / facing_sector | Источник цели или направления. |
| `targetPoint` | `null` | координаты X/Y | Явная точка цели, если она задана автором графа. |
| `sectorCenterDegrees` | `0°` | `-360…360°` | Смещение центра сектора относительно взгляда бойца. |
| `sectorArcDegrees` | `90°` | `1…360°` | Ширина сектора поиска. |
| `searchRadiusMeters` | `50 м` | `1…500 м` | Область чтения статического индекса и поле достижимости. |
| `maxCandidates` | `12` | `1…16` в редакторе | Максимум готовых позиций, возвращаемых графу. |
| `desiredDistanceMeters` | `0 м` | `0…3000 м` | Желаемая дистанция. Ноль берёт дистанцию цели или оружия. |

## Общий баланс оценки

| Поле | По умолчанию | Диапазон | Большое значение |
|---|---:|---:|---|
| `tacticalQualityWeight` | defense `0.58`; observation/firing `0.66` | `0…10` | Сильнее учитывает качество самой позиции. |
| `movementObjectiveWeight` | defense `0.42`; observation/firing `0.34` | `0…10` | Сильнее учитывает выбранную задачу движения. |

Значение `movementObjectiveWeight = 0` полностью отключает влияние задачи движения, включая запасной критерий сортировки. Поэтому `advance_to_threat` и `withdraw_from_threat` при нулевом весе дают одинаковый порядок для одинаковых по качеству позиций.

## Веса составляющих качества

Все веса имеют диапазон `0…10` и шаг `0.01`. Ноль выключает составляющую. Формула использует нормированное взвешенное среднее, поэтому важны отношения между весами, а не их абсолютная сумма.

| Поле | observation | defense | firing | Смысл |
|---|---:|---:|---:|---|
| `staticPotentialWeight` | `0.19` | `0.18` | `0.16` | Качество постоянной основы места. |
| `directionalFitWeight` | `0.13` | `0.16` | `0.12` | Соответствие нужному направлению. |
| `lineQualityWeight` | `0.25` | `0.03` | `0.24` | Точная линия наблюдения, защиты или огня. |
| `rangeFitWeight` | `0` | `0` | `0.16` | Соответствие рабочей дальности оружия. |
| `desiredDistanceWeight` | `0` | `0` | `0` | Соответствие явно желаемой дистанции. |
| `protectionWeight` | `0.08` | `0.29` | `0.08` | Защита от известного направления угрозы. |
| `concealmentWeight` | `0.12` | `0.09` | `0.06` | Скрытность позиции. |
| `dangerWeight` | `0.09` | `0.10` | `0.07` | Предпочтение низкой опасности самой позиции. |
| `routeDangerWeight` | `0.05` | `0.06` | `0.04` | Предпочтение безопасного пути. |
| `routeCostWeight` | `0` | `0` | `0` | Предпочтение дешёвого и короткого пути. |
| `certaintyWeight` | `0.04` | `0.04` | `0.02` | Предпочтение данных с низкой неопределённостью. |
| `orderAlignmentWeight` | `0.03` | `0.03` | `0.03` | Близость к точке приказа. |
| `withdrawalWeight` | `0.02` | `0.02` | `0.02` | Качество безопасного пути отхода. |
| `postureFitWeight` | `0` | `0` | `0` | Качество рекомендуемой позы и цена перехода. |

Старые графы получают именно эти значения. Старые коэффициенты были перенесены без изменения результата: скрытые `58/42` теперь являются обычными полями ноды.

## Задача движения

| Поле | По умолчанию | Диапазон | Назначение |
|---|---:|---:|---|
| `balancedObjectiveInfluence` | `0` | `0…10` | Сила сохранения текущей дистанции в balanced. Ноль сохраняет старое нейтральное поведение. |
| `advanceObjectiveInfluence` | `1` | `0…10` | Сила приближения к угрозе. |
| `withdrawObjectiveInfluence` | `1` | `0…10` | Сила удаления от угрозы. |
| `continueOrderObjectiveInfluence` | defense `0.5918367347`; observation/firing `0.8319327731` | `0…10` | Сила приближения к точке приказа. Значения сохраняют старое эффективное отношение около `70/30`. |
| `wrongDirectionPenalty` | `0%` | `0…100%` | Дополнительный штраф за движение вопреки задаче. |
| `objectiveDistanceToleranceMeters` | `2 м` | `0…100 м` | Изменение дистанции внутри допуска считается нейтральным. |

Итоговый порядок меняется в одном каноническом ранжировщике. Отдельной скрытой пересортировки после точного поиска нет.

## Ограничения пригодности

Ограничения отбрасывают позицию до итогового мягкого ранжирования.

| Поле | По умолчанию | Диапазон | Поведение |
|---|---:|---:|---|
| `maxPositionDanger` | `78%` | `0…100%` | Позиции опаснее порога не участвуют. |
| `maxRouteDanger` | `100%` | `0…100%` | Путь опаснее порога не допускается. |
| `minimumProtection` | defense `10%`; остальные `0%` | `0…100%` | Минимальная защита. |
| `minimumConcealment` | `0%` | `0…100%` | Минимальная скрытность. |
| `minimumDirectionalFit` | observation/firing `18%`; defense `0%` | `0…100%` | Минимальное соответствие нужному направлению. |
| `minimumLineQuality` | `18%` | `0…100%` | Минимальная точная линия наблюдения, защиты или огня. |
| `minimumTargetDistanceMeters` | `0 м` | `0…3000 м` | Минимальная дальность до цели. |
| `maximumTargetDistanceMeters` | `0 м` | `0…5000 м` | Максимальная дальность; ноль использует цель или оружие. |
| `desiredDistanceToleranceMeters` | `10 м` | `0.1…1000 м` | Область около желаемой дистанции без сильного штрафа. |
| `requireVisualLine` | `false` | да/нет | Для наблюдения требует выполненную точную визуальную проверку. |
| `requireBallisticLine` | `false` | да/нет | Для огневой позиции требует выполненную точную баллистическую проверку. |

Если минимум больше ненулевого максимума дистанции, загрузчик поднимает максимум до минимума. `NaN`, `Infinity` и отрицательные значения заменяются или ограничиваются безопасным диапазоном.

## Поза

| Поле | По умолчанию | Диапазон | Назначение |
|---|---:|---:|---|
| `allowStanding` | `true` | да/нет | Разрешить позицию стоя. |
| `allowCrouched` | `true` | да/нет | Разрешить позицию пригнувшись. |
| `allowProne` | `true` | да/нет | Разрешить позицию лёжа. |
| `transitionPenaltyStanding` | `3` | `0…50` | Цена перехода в стойку. |
| `transitionPenaltyCrouched` | `3` | `0…50` | Цена перехода в пригнувшуюся позу. |
| `transitionPenaltyProne` | `7` | `0…50` | Цена перехода лёжа. |
| `postureDangerExposureWeight` | `0.30` | `0…10` | Насколько опасность понижает более открытую позу. |

Нельзя отключить все три позы: нормализатор безопасно возвращает `standing = true`. Это предотвращает запрос, который заведомо не может вернуть ни одной позиции.

## Производительность

Эти поля доступны в свёрнутой расширенной секции. Их завышение повышает время поиска и объём памяти, но не снимает жёсткие пределы.

| Поле | По умолчанию | Безопасный диапазон | Влияние |
|---|---:|---:|---|
| `candidateScanLimit` | `864` | `64…4096` | Сколько статических мест разрешено прочитать из индекса. |
| `preliminaryCandidates` | `36` | `8…128` | Сколько лучших дешёвых кандидатов оставить. |
| `exactCandidates` | `12` | `1…32` | Сколько кандидатов подвергать точным проверкам. |
| `exactRayLimit` | `32` | `0…128` | Общий предел точных лучевых проверок. |
| `maxRouteExpansions` | `1728` | `64…8192` | Предел расширений единого поля достижимости. |
| `maximumRouteCost` | `100000` | `1…1000000` | Предел стоимости достижимого пути. |
| `objectiveCandidatePool` | `12` | `1…32` | Размер окончательной выборки перед ранжированием задачи движения. |
| `minimumSeparationMeters` | `4 м` | `0…100 м` | Минимальный разнос похожих итоговых позиций. |
| `maxCalculationMs` | `12 мс` | `0.1…100 мс` | Совместимый бюджет старого синхронного шага графа. Тяжёлый поиск ограничивается числовыми бюджетами выше. |

Нормализатор поддерживает зависимости бюджетов:

- точных кандидатов не меньше числа итоговых результатов;
- предварительных кандидатов не меньше точных;
- лимит просмотра не меньше предварительной выборки;
- окончательная выборка не меньше числа результатов и не больше точной выборки.

Поиск строит одно ограниченное поле достижимости. A* для каждого кандидата не запускается.

## Путь параметров

```text
панель свойств ноды
→ плоские параметры JSON Graph v2
→ readTacticalPositionNodeSettings
→ AiGraphRunner
→ SimulationTacticalPositionGraphHost
→ TacticalPositionSearchService
→ target.nodeSearchSettings в worker request
→ ConfiguredGeneralizedTacticalPositionSearch
→ предварительное ранжирование
→ точные проверки
→ каноническое итоговое ранжирование
→ TacticalPositionCandidateSeedV2
```

Плоский JSON выбран ради совместимости Graph v2. После чтения он один раз превращается во вложенный типизированный объект `TacticalPositionSearchSettings` с группами `ranking`, `movementObjective`, `constraints`, `posture` и `searchBudget`.

Настройки передаются внутри сериализуемой оболочки цели. Они не содержат `ArrayBuffer`, поэтому не входят в список переносимых буферов и не могут их отсоединить.

## Идентичность и устаревание

- Digest всех субъективных настроек входит в идентичность конфигурации графа.
- Сохранённый `requestId` переиспользуется только при совпадении digest.
- Изменение веса, порога, позы или бюджета создаёт новый запрос.
- Устаревший ответ не подменяет результат новой конфигурации.
- `StaticTacticalPositionBasis` не импортирует настройки ноды и не включает их в digest основы.

## Совместимость

- Старый граф без новых полей получает канонические значения по умолчанию при чтении.
- Неизвестные допустимые поля не мешают загрузке и сохраняются нормализатором.
- Клонирование ноды копирует плоские параметры обычным механизмом редактора.
- `CreateCoverCandidates` не удалена. Раннер по-прежнему принимает её как совместимый оборонительный запрос с задачей `balanced`.
- Обязательной миграции сохранений нет.
- Старые веса типов сохранены в начальных значениях.

## Параметры, оставленные внутренними

| Параметр | Причина |
|---|---|
| Версия worker-протокола, `jobId`, `basisIdentityKey`, поля stale/cancelled | Служебная целостность асинхронного обмена. Их изменение не описывает тактическое поведение. |
| Список переносимых `ArrayBuffer` | Владение памятью рабочего потока. Ошибочная настройка повреждает данные. |
| Число секторов постоянной основы, длины и шаг статических лучей, плотность образцов, размер частей карты и пороги статического индекса | Меняют общую постоянную основу и требуют её тяжёлой перестройки. Это настройка карты/движка, а не отдельного запроса бойца. |
| Физические высоты и относительная открытость standing/crouched/prone | Калибровка размеров тела и геометрии луча. Пользователь управляет допустимыми позами, штрафами перехода и силой влияния опасности, но не меняет физический размер бойца из ноды. |
| Диагональная цена `sqrt(2)` и внутренний перевод опасности в стоимость поля достижимости | Единицы физического маршрутизатора. Для тактического выбора доступны `routeCostWeight`, `routeDangerWeight`, `maxRouteDanger`, `maximumRouteCost` и бюджет расширений. |
| Слияние геометрической и субъективной защиты `56/44` для defense и `72/28` для observation/firing | Калибровка значения одного сенсорного показателя, а не предпочтение позиции. Итоговая важность защиты полностью задаётся `protectionWeight` и `minimumProtection`. |
| Порог сходства направлений при удалении дублей | Инвариант разнообразия выдачи и устойчивого идентификатора кандидата. Пользователь управляет пространственным разносом через `minimumSeparationMeters`. |
| Коэффициенты диагностического поля `safety` и пороги подписи склона | Используются только в диагностике и не входят в итоговый выбор. |
| Детерминированные правила разрешения равенства | Нужны для воспроизводимости и не выражают тактическое предпочтение. |

## Пресеты

Пресеты не добавлены. В текущем редакторе нет общего механизма пресетов, а отдельная реализация только для этой ноды создала бы второй путь значений и дополнительное состояние `custom`. Все значения доступны явно, поэтому безопаснее сначала использовать обычное клонирование нод и групповой сброс.

## Практические рекомендации

- Для осторожного поведения повышайте `dangerWeight`, `routeDangerWeight`, `protectionWeight`, уменьшайте `maxPositionDanger` и `maxRouteDanger`.
- Для напора повышайте `movementObjectiveWeight` и `advanceObjectiveInfluence`; не повышайте одновременно жёсткие минимумы защиты и скрытности, иначе подходящих мест может не остаться.
- Для огневой позиции сначала задайте `rangeFitWeight`, `desiredDistanceWeight` и при необходимости `requireBallisticLine`.
- Для наблюдения повышайте `lineQualityWeight` и `concealmentWeight`; `requireVisualLine` увеличивает цену точных проверок.
- `preliminaryCandidates`, `exactCandidates`, `exactRayLimit` и `maxRouteExpansions` повышайте последовательно. Сначала увеличивайте предварительную выборку, затем точную, затем лучи.
- Не ставьте большие бюджеты без измерения. Полный перебор карты намеренно невозможен.
