# Graph v2: типизированные контракты и подграфы

Дата: 2026-07-13  
Ветка переноса: `transfer/ai-graph-v2-preview-2026-07-13`  
Статус: чистое transfer-дерево подготовлено поверх `real-wargame-preview` `db80f36edaf018c6a45dfeb7cc0f7caaed00bdb5`; перед слиянием требуется точная CI/Chrome-проверка ветки.

## Что изменилось простыми словами

Graph v2 делает редактор поведения безопаснее. Теперь редактор знает не только название ноды, но и какие данные она принимает, что возвращает и какие параметры разрешены.

## Контракт авторства будущего боевого поведения

Это архитектурное требование к следующим этапам, а не утверждение о том, что полный боевой цикл уже реализован.

Итоговое поведение одиночного бойца должно собираться и настраиваться через Graph v2 и редактор нод с самого начала. Не допускается сначала реализовать полный отдельный вариант поведения жёстко в TypeScript, а затем переносить его на ноды: такой путь создаёт две расходящиеся реализации.

Разделение ответственности фиксируется следующим образом.

**Детерминированный код и источники фактов** владеют:

- восприятием, субъективной памятью угроз и математикой опасности;
- сохранением и затуханием фактического подавления;
- расчётами укрытия, маскировки, леса, рельефа, дальности и точности оружия;
- поиском пути, владением движением и применением маршрута;
- проверкой линии огня, боеприпасов и дружественного огня;
- сериализацией runtime и детерминированным жизненным циклом действий.

**Graph v2 и подграфы** владеют политикой и композицией решений:

- порогами, весами, задержками, deadband и гистерезисом;
- Utility-выбором реакции;
- переходами между оценкой обстановки, реакцией, укрытием, позой, выбором цели, огнём и возвращением к приказу;
- условиями начала, продолжения, переноса, прекращения и возобновления огня.

Защитные инварианты остаются в коде и не могут отключаться или обходиться нодами:

- никакого чтения скрытых объективных данных противника;
- очистка движения и действий только при доказанном `ownerToken`;
- запрет выстрела без боеприпасов или безопасной линии огня;
- проверка риска для своих;
- ровно один детерминированный `start / update / complete / cancel`;
- запрет запуска A* каждый кадр.

Безопасные значения по умолчанию могут задаваться контрактом ноды или схемой в коде. Их поведенческая часть должна переопределяться графом. Настраиваемые параметры должны:

- быть доступны в свойствах ноды без редактирования JSON или TypeScript;
- сериализоваться и восстанавливаться вместе с графом;
- иметь полные русские названия, подсказки и причины;
- отображаться в trace вместе с фактическим значением и его влиянием на решение.

Поведенческое число не считается закрытым критерием среза, если оно осталось неэкспонированной константой, хотя пользователь должен иметь возможность его настраивать.

Срезы 1–4 кампании Stage 1–2 предоставляют графу чистые providers, query-ноды и диагностику фактов. Срезы 5–9 собирают через Graph v2 и подграфы само поведение, включая `AssessCombatSituation`, `ReactToSuddenFire`, `FindAndOccupyCover`, `ChoosePosture`, `ResumeOrder`, приоритет целей и полный жизненный цикл огня.

### Типизированный порт

Порт — это подписанный вход или выход ноды. У каждого порта есть тип, например:

- число;
- да/нет;
- текст;
- позиция;
- боец;
- событие;
- маршрут.

Редактор не позволяет соединить несовместимые типы. Например, идентификатор бойца нельзя передать во вход позиции. Причина показывается по-русски до сохранения неправильной связи.

### Контракт ноды

Контракт — единое описание ноды для core и редактора. В нём хранятся:

- русское и английское название;
- подробное описание;
- входы и выходы;
- обязательные и необязательные параметры;
- значения по умолчанию;
- диапазоны и варианты выбора;
- допустимое число дочерних нод;
- тип жизненного цикла.

Благодаря этому значения по умолчанию и подсказки больше не дублируются отдельно в интерфейсе и runtime.

## Миграция старых графов

Graph v1 продолжает загружаться. В памяти он детерминированно преобразуется в Graph v2. Порядок детей сохраняется, старые параметры получают типы, а неизвестные поля переносятся в `legacyMetadata`.

В редакторе есть действие:

```text
Проверить и обновить формат графа
```

Graph v2 сохраняется только после успешной миграции и проверки. При невозможной миграции старый граф не перезаписывается, а пользователь получает русскую причину.

## Области памяти

Память разделена на пять уровней:

1. **Постоянная память бойца** — знания, которые должны жить долго.
2. **Память runtime session** — состояние текущего сеанса исполнения.
3. **Память активного состояния** — данные текущей выполняемой ветви.
4. **Локальная память подграфа** — изолированные данные одного вызова подграфа.
5. **Локальное состояние ноды** — внутренний прогресс конкретной ноды.

Локальная память подграфа не изменяет память родителя без явной выходной привязки. Debug-панель показывает область и ключи памяти.

## Подграфы

Подграф — переиспользуемый фрагмент поведения. Родительский граф передаёт ему только явно указанные входы и получает только явно указанные выходы.

Доступны четыре шаблона:

- `take_cover` — «Занять укрытие»;
- `reload_weapon` — «Перезарядить оружие»;
- `react_to_fire` — «Реагировать на обстрел»;
- `move_and_observe` — «Двигаться и наблюдать».

В русской панели «Человеческий интерфейс ноды» видны:

- выбор шаблона поведения;
- политика отмены;
- реальные входы выбранного подграфа;
- реальные выходы выбранного подграфа.

Подграф открывается двойным щелчком. Навигационная цепочка показывает путь вида:

```text
Главный граф → Реагировать на обстрел → Занять укрытие
```

## Runtime и сохранение

Активный подграф сохраняется внутри runtime session. После загрузки длительное действие продолжает `update`, а не получает повторный `start`. Отмена родителя отменяет активный подграф, а cleanup выполняется ровно один раз.

След исполнения содержит полный путь:

```text
main_graph / take_cover / move_to_cover
```

## Управляющие ноды

Перед Graph v2 закрыты недостающие части этапа 2:

- `WaitForEvent` — ждёт указанное событие и при необходимости поглощает его;
- `Timeout` — отменяет дочернее действие после лимита времени;
- `Retry` — повторяет дочерний шаг не больше заданного числа попыток.

Их состояние сериализуется вместе с runtime.

## Контрольный автоматический сценарий

Проверяется последовательность:

```text
приказ движения
→ MoveTo Running
→ shot_nearby
→ реактивная отмена старого движения
→ take_cover
→ сохранение во время движения
→ восстановление без повторного start
→ достижение укрытия
→ move_and_observe
```

Проверка подтверждает защиту `ownerToken`, единственный cleanup, изоляцию памяти, восстановление подграфа и отсутствие зависшего runtime handle после завершения.

## Что проверить вручную после переноса

1. Открыть редактор ИИ и увидеть предупреждение Graph v1.
2. Нажать «Проверить и обновить формат графа».
3. Создать ноду «Подграф поведения».
4. В русской панели выбрать каждый из четырёх шаблонов и проверить изменение списка входов/выходов.
5. Попробовать несовместимое соединение и увидеть русскую причину запрета.
6. Дважды щёлкнуть по подграфу, проверить цепочку навигации и возврат.
7. Нажать на ошибку в панели проверки и убедиться, что выделяется нужная нода.
8. В живом debug проверить активный подграф, путь исполнения и пять областей памяти.
