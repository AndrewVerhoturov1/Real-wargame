# Обзор и память — единая система видимости

## Назначение

Слой **«Обзор и память»** показывает субъективные возможности выбранного бойца:

- тепловая карта отображает качество текущего наблюдения местности;
- текущие и старые контакты показываются отдельными метками;
- старая тепловая карта местности не сохраняется как память;
- слой не раскрывает скрытых бойцов, объектов сценария или других неизвестных источников.

Расчёт физической видимости теперь имеет одно каноническое математическое ядро. Его используют:

1. фактическое восприятие конкретной цели;
2. тепловая карта выбранного бойца;
3. диагностический инструмент проверки линии видимости.

Рендерер только показывает готовый результат. Владельцем субъективного обнаружения остаётся `PerceptionSystem`.

## Каноническое ядро луча

`VisibilityRayKernel` проводит точный отрезок между фактическими координатами наблюдателя и цели.

Ядро учитывает:

- точные координаты `x/y`, без переноса бойцов в центр клетки;
- плавную высоту рельефа под наблюдателем и целью;
- высоту глаз наблюдателя с учётом позы;
- высоту проверяемой точки цели;
- горизонт рельефа;
- растеризованные высоты блокирующих объектов;
- точную длину пути внутри каждой клетки растительности;
- отдельную проходимость для обзора и огня;
- край карты.

Пересечение сетки рассчитывается детерминированным DDA-проходом. Длина растительности внутри клетки больше не заменяется постоянным шагом или условным осевым/диагональным множителем:

```text
прохождение *= exp(-потеря_на_метр × фактическая_длина_пути)
```

Если накопленная растительность опускает визуальную проходимость ниже минимального порога материала, оставшаяся часть визуального луча считается закрытой.

## Положение и поза наблюдателя

Высота глаз человека задаётся одной канонической таблицей:

| Поза | Высота глаз |
|---|---:|
| Стоит | 1,70 м |
| Пригнулся | 1,10 м |
| Лежит | 0,35 м |

Координаты наблюдателя используются в их фактическом состоянии. Перемещение внутри одной клетки может изменить линию обзора и создаёт новую геометрическую идентичность кеша.

## Фактическая цель

Для существующего бойца восприятие получает из `PerceptionStimulus`:

- фактическое положение цели;
- фактическую позу;
- соответствующую высоту силуэта;
- движение, действие, размер и маскировку.

Цель не проверяется одной условной точкой. `VisibilityTargetProbe` проводит три канонических луча:

- нижняя часть тела — `30%` высоты силуэта;
- корпус — `60%`;
- верхняя часть и голова — `90%`.

Заблокированный уровень даёт нулевой вклад. Открытый уровень передаёт фактическую проходимость своего луча:

```text
средняя проходимость силуэта =
  (нижняя + корпус + верхняя) / 3
```

Результат содержит:

- число видимых уровней от `0` до `3`;
- видимую долю силуэта;
- среднюю визуальную проходимость;
- лучший открытый уровень;
- причины перекрытия каждого уровня.

Поэтому система различает полностью скрытую цель, частично показавшийся корпус или голову и полностью открытого бойца. Частичный силуэт уменьшает накопление визуальных признаков, а не превращается автоматически в полную видимость.

## Тепловая карта

Пустая клетка не имеет реальной цели и позы. Слой никогда не читает скрытых бойцов, чтобы определить её цвет.

В панели есть отображаемый режим условной цели:

- **Стоит** — 1,70 м;
- **Пригнулся** — 1,10 м;
- **Лежит** — 0,35 м.

Это только диагностическая высота для тепловой карты. Переключатель:

- не меняет позу выбранного бойца;
- не меняет цели ИИ;
- не создаёт и не обновляет контакты;
- не раскрывает наличие противника в клетке.

По умолчанию используется режим **«Стоит»**.

## Маска кандидатов

Вся карта изначально считается тенью. Перед геометрией строится дешёвая маска возможного обзора.

Для клетки проверяются только:

1. расстояние;
2. направление относительно фокуса;
3. зона внимания;
4. вес зоны;
5. максимальная дальность этой зоны.

Клетка вне разрешённой зоны:

- остаётся `unseen`;
- получает качество `0`;
- не получает целевой геометрический LOS;
- остаётся стандартной тенью.

Луч к разрешённой клетке всё равно может проходить через клетки вне внимания: они нужны как физический рельеф, объект или растительность, но не оцениваются как наблюдаемые цели.

Пустая маска создаёт полностью затенённое поле с нулём геометрических лучей.

## Качество наблюдения клетки

Для разрешённой и геометрически проверенной клетки качество вычисляется так:

```text
качество =
  коэффициент дистанции
  × визуальная проходимость
  × вес внимания
  × состояние наблюдателя
```

На состояние наблюдателя влияют здоровье, усталость, замешательство и подавление.

Геометрическое поле отвечает только за физическую возможность наблюдения. Поза, движение, стрельба, относительный размер и маскировка реальной цели дополнительно применяются в `VisualSignal`.

## Внимание и расписание

Дорогая проверка конкретной цели выполняется только когда:

- цель попала в широкую фазу по дальности;
- направление входит в существующую зону внимания;
- для этой зоны наступил очередной интервал проверки.

Одновременно сохраняется ограничение:

```text
не более 2 новых логических проверок цели за шаг симуляции
```

Одна логическая проверка цели выполняет ровно три физических луча силуэта. Повторный запрос с теми же точными координатами, позой, высотой, ревизиями карты и профилем среды берётся из ограниченного кеша.

## Кеширование и перестроение тепловой карты

Поле выбранного бойца перестраивается при изменении значимых данных:

- фактической позиции;
- позы наблюдателя;
- направления или режима внимания;
- параметров зон;
- параметров зрения;
- состояния наблюдателя;
- условной позы цели для слоя;
- визуальной ревизии карты;
- профиля растительности.

Во время движения перестроение ограничено интервалом около `0,2` секунды. Между обновлениями используется последнее завершённое поле.

Движение камеры и курсора не входит в ключ и не запускает расчёт.

## Нулевые расходы скрытого слоя

Тепловая карта не строится, когда:

- слой выключен;
- выключен переключатель текущего обзора;
- боец не выбран;
- включён редактор карты.

Фактическое восприятие других бойцов при этом продолжает работать по своему расписанию и не зависит от видимости пользовательского слоя.

## Отображение

PixiJS не создаёт объект на каждую клетку. Локальные `Uint8Array` переводятся в одну растровую текстуру и показываются одним `Sprite`.

Текстура загружается только при изменении ревизии поля. Метки контактов и памяти рисуются поверх отдельным `Graphics`.

## Диагностика

Панель показывает:

- количество перестроений и попаданий в кеш;
- причину последнего обновления;
- число клеток-кандидатов;
- число целевых клеток, проверенных геометрией;
- число клеток, пройденных лучами;
- число лучей поля.

Диагностика точечных проверок отдельно хранит:

- число логических проверок целей;
- число физических лучей силуэта;
- отложенные проверки из-за бюджета;
- попадания в кеш.

## Ограничения

- Тепловая карта строится только для выбранного бойца и только по запросу интерфейса.
- Три уровня силуэта являются тактической моделью, а не анатомической симуляцией.
- Геометрия объектов пока использует единый растеризованный слой карты. Для более точной формы повёрнутых объектов нужно улучшать этот общий слой, а не создавать отдельный LOS для восприятия.
- Оптика, ночь, подробная погода и командный обмен контактами не входят в эту версию.
- Тепловая карта остаётся диагностическим пользовательским представлением и не является источником объективного знания для ИИ.
