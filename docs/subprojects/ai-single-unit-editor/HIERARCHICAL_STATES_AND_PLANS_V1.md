# Иерархические состояния и планы ИИ v1

Дата: 2026-07-14  
Ветка разработки: `feat/ai-state-plan-v1-temp-2026-07-14`  
База: `real-wargame-preview` `cc907ca0f48caed418cd76b0f878c8b18fbe71c7`

## Зачем появился этот слой

До этого Graph v2 хорошо исполнял отдельные длительные действия: движение, ожидание, наблюдение и подграфы. Но одному графу было трудно ответить на более общий вопрос: **в какой общей ситуации сейчас находится боец и какую цель он должен удерживать несколько шагов подряд**.

Новый слой разделяет четыре разных задачи:

```text
Общее состояние бойца
→ ограничивает допустимые варианты

Utility AI
→ выбирает лучший допустимый вариант

AiPlan
→ закрепляет цель и последовательность шагов

Graph v2 / подграф
→ исполняет текущий конкретный шаг

Action Runtime
→ реально двигает бойца, меняет позу и ждёт события
```

## Что такое общее состояние

**Состояние** описывает не отдельное действие, а общую ситуацию бойца.

В первой версии есть четыре состояния:

| Состояние | Русское название | Смысл |
|---|---|---|
| `Idle` | Ожидание | Нет активного приказа движения и боевой чрезвычайной ситуации. |
| `FollowingOrder` | Выполнение приказа | Боец выполняет обычный приказ движения. |
| `Contact` | Контакт | Замечен противник или получено событие боевого контакта. |
| `Suppressed` | Подавлен | Подавление достигло критического уровня, поэтому самосохранение важнее обычного приказа. |

Состояния объединены в два родителя:

```text
Обычное состояние / Normal
├─ Ожидание / Idle
└─ Выполнение приказа / FollowingOrder

Бой / Combat
├─ Контакт / Contact
└─ Подавлен / Suppressed
```

При переходе `Контакт → Подавлен` общий родитель `Бой` остаётся активным. Runtime выходит только из `Contact` и входит только в `Suppressed`. Это не перезапускает весь боевой контекст без необходимости.

## Чем состояние отличается от ноды

Нода отвечает на узкий вопрос или выполняет один элемент поведения:

- проверить порог;
- найти укрытие;
- начать движение;
- ждать событие;
- переключить позу.

Состояние отвечает на общий вопрос:

> Какие виды поведения вообще допустимы сейчас?

Например, в `Suppressed` обычное выполнение приказа не участвует в Utility-выборе на равных с самосохранением. Разрешены только занятие укрытия, смена позы и ожидание снижения подавления.

## Как происходят переходы состояний

Основные переходы первой версии:

```text
Ожидание + получен приказ движения
→ Выполнение приказа

Выполнение приказа + замечен противник
→ Контакт

Любое состояние + критическое подавление
→ Подавлен

Подавлен + подавление устойчиво снизилось
→ Контакт

Выполнение приказа + приказ завершён или отменён
→ Ожидание
```

Если одновременно подходят несколько переходов:

1. побеждает больший приоритет;
2. при одинаковом приоритете используется стабильный порядок по `transition.id`;
3. результат не зависит от порядка объектов в памяти или от FPS.

Для защиты от дрожания используются:

- минимальное время в состоянии;
- отдельный более низкий порог выхода из подавления;
- время, в течение которого подавление должно устойчиво оставаться ниже этого порога;
- экстренный wildcard-переход `* → Suppressed`, который может прервать обычное поведение.

Каждый переход записывает:

- старое и новое состояние;
- причину на английском и русском;
- trigger;
- время симуляции;
- какие уровни дерева были покинуты и активированы.

## Что такое план

**План** — это сохраняемый объект с одной удерживаемой целью и последовательностью шагов.

Он содержит:

- уникальный `plan.id`;
- цель на английском и русском;
- состояние, для которого план был создан;
- список шагов;
- текущий шаг и число попыток;
- статус;
- причины выбора;
- условия отмены и перестроения;
- ссылку на заменённый план.

В первой версии есть два плана.

### FollowMoveOrder — выполнить приказ движения

```text
Цель: выполнить приказ движения

Шаг 1:
Двигаться и наблюдать
подграф: move_and_observe
```

План прекращается или перестраивается, когда:

- приказ отменён;
- получен новый приказ;
- появился боевой контакт;
- маршрут окончательно заблокирован.

### TakeCover — занять укрытие

```text
Цель: занять укрытие

Шаг 1:
Движение к укрытию
подграф: take_cover

Шаг 2:
Наблюдение после занятия позиции
подграф: move_and_observe
```

План прекращается или перестраивается, когда:

- укрытие стало недоступно;
- маршрут к укрытию заблокирован;
- найдено другое, более безопасное укрытие;
- более приоритетный переход состояния делает план недопустимым.

План `TakeCover` допустим и в `Contact`, и в `Suppressed`. Поэтому переход `Contact → Suppressed` не обязан начинать второе движение: существующий план занятия укрытия может продолжиться.

## Чем план отличается от подграфа

Подграф — это повторно используемый исполнитель одного конкретного шага.

План — это более высокий уровень:

- удерживает цель между тиками;
- знает текущий шаг;
- знает, что делать после успеха или неудачи;
- умеет отменяться и заменяться;
- сохраняется вместе со сценой;
- объясняет пользователю, почему выбран.

Пример:

```text
План TakeCover
├─ подграф take_cover
└─ подграф move_and_observe
```

Подграфы не дублируются и не получают отдельный параллельный механизм движения. План создаёт маленький служебный Graph v2 текущего шага, который вызывает существующий зарегистрированный подграф. Движение по-прежнему принадлежит существующему Action Runtime и защищено `ownerToken`.

## Как связаны State → Utility → Plan → Subgraph

1. `AiStateRuntime` получает события приказа, контакта и текущий уровень подавления.
2. Состояние определяет разрешённые Utility-ветви.
3. Пока активный план остаётся допустимым и валидным, Utility не пересчитывается каждый тик.
4. Когда нужен новый выбор, `AiStatePlanPipeline` выбирает лучший разрешённый вид плана.
5. `AiPlanRuntime` переводит первый ожидающий шаг в `running` ровно один раз.
6. Служебный Graph v2 вызывает подграф шага.
7. Существующий Action Runtime продолжает `update` до успеха, отказа или отмены.
8. После успеха план переходит к следующему шагу или завершается.

## Как происходит отмена

Экстренный переход состояния выполняется в строгом порядке:

```text
1. Новое состояние признало старый план недопустимым.
2. Активный Graph v2 получает cancel.
3. Вложенный подграф передаёт cancel активной action-нoded.
4. Старое движение выдаёт ровно один clear_move с прежним ownerToken.
5. Старый план получает cancelled или replanning и сохраняется в истории.
6. Только после этого разрешено создать новый план.
7. Новый план создаёт новый ownerToken движения.
```

Так старый cleanup не может удалить новый приказ, и у бойца не появляется два активных владельца движения.

## Как происходит перепланирование

Перепланирование отличается от простого повторного запуска:

- старый план сохраняет свою историю и причину;
- старый активный подграф корректно отменяется;
- новый план получает новый `id`;
- поле `replacesPlanId` указывает, какой план он заменил;
- число попыток старого шага не переносится в новый план;
- новый шаг запускается только один раз.

## Что сохраняется в сцене

Runtime snapshot теперь хранит:

- активное leaf-состояние;
- полный путь родителей;
- предыдущее состояние;
- время входа;
- последнюю причину перехода и trace;
- активный план;
- текущий шаг;
- статусы и число попыток шагов;
- историю завершённых, отменённых и перестроенных планов;
- `replacesPlanId`;
- вложенное состояние выполняемого подграфа;
- токен владельца вложенного движения.

После загрузки:

- состояние не вызывает повторный `enter`;
- план не создаётся заново;
- шаг со статусом `running` не вызывает повторный `start`;
- вложенный подграф продолжает `update`;
- движение не получает второй маршрут;
- cleanup не вызывается второй раз.

Старая сцена без новых полей загружается безопасно:

```text
state = Idle
activePlan = null
```

## Что видит пользователь

В нижней панели выбранного бойца появился компактный раскрываемый блок:

- текущее состояние;
- родительское и предыдущее состояние;
- понятная причина перехода;
- активный план и его статус;
- текущий шаг и номер шага;
- причины выбора;
- условия отмены и перестроения;
- технические id только в отдельной диагностике.

В редакторе нод отдельная живая панель читает тот же runtime debug payload и показывает состояние, план и активный подграф. Кнопка **«Показать активный подграф»** открывает соответствующий зарегистрированный подграф в редакторе.

Панели создаются один раз. На тиках обновляются только изменившиеся текстовые значения, классы и короткие списки; весь DOM не перестраивается.

## Границы версии 1

В эту версию намеренно не входят:

- раненый, отступающий и паникующий боец;
- полноценная мораль;
- командирские состояния;
- стрельба и подавляющий огонь как новые планы;
- Tactical Query System;
- параллельные планы.

Система стрельбы может добавлять свои события, условия и планы поверх этого контракта, но не должна обходить единый порядок `State → Utility → Plan → Subgraph → Action Runtime`.
