# Reactive Abort + Route Status v1

Дата: 2026-07-12  
Подпроект: `ai-single-unit-editor`  
Статус: реализовано во временной ветке, **не перенесено в `real-wargame-preview`**

Рабочая ветка:

```text
feature/ai-reactive-route-status-v1
```

## Главное изменение

`MoveToBlackboardPosition` теперь не просто ждёт прибытия или общего тайм-аута. Система отдельно наблюдает, действительно ли боец приближается к своей замороженной цели.

Она различает:

```text
Движение
Нет прогресса
Маршрут заблокирован
Цель достигнута
Новый приказ игрока
Цель потеряна
Приказ движения исчез
```

При значимом событии runtime пересчитывается немедленно, а не ждёт следующего обычного интервала ИИ.

## Важная честная граница

В этой версии **нет pathfinding**.

`Маршрут заблокирован` означает:

> расстояние до цели не уменьшается достаточно долго.

Система пока не знает, какое именно препятствие мешает и как его обойти. Она фиксирует измеряемое застревание и корректно прерывает действие. Это фундамент для будущего поиска пути, но не его имитация.

Существующая физика остаётся прямолинейной:

```text
SimulationTick
→ двигает бойца напрямую к target
```

## Новый чистый модуль

Добавлен:

```text
src/core/ai/AiRouteStatus.ts
```

Он не импортирует:

- DOM;
- PixiJS;
- localStorage;
- `SimulationState`;
- игровой интерфейс.

Модуль получает снимок положения, цели и приказа, затем возвращает новое serializable-состояние маршрута и сигнал для bridge.

## Состояния маршрута

```text
idle             — активного маршрута нет
moving           — есть измеряемое продвижение
stalled          — продвижение временно не обнаружено
blocked          — лимит отсутствия прогресса превышен
arrived          — цель достигнута
player_override  — игрок заменил приказ ИИ
 target_lost     — ключ цели исчез из Blackboard
order_missing    — собственный приказ движения исчез
cancelled        — маршрут отменён
```

В коде `target_lost` записывается без начального пробела; пробел выше не является частью имени статуса.

## Алгоритм прогресса

При старте сохраняются:

```text
ownerToken
замороженная цель
время старта
время последней проверки
время последнего заметного прогресса
последнее контрольное расстояние
текущий статус
```

На каждом лёгком bridge-полле:

1. Измеряется расстояние до замороженной цели.
2. Если цель достигнута — `arrived`.
3. Если игрок заменил приказ — `player_override`.
4. Если собственный приказ исчез — `order_missing`.
5. Если ключ цели исчез и настройка включена — `target_lost`.
6. Если расстояние уменьшилось достаточно — `moving`, таймер сбрасывается.
7. Если прогресса пока недостаточно — `stalled`.
8. Если лимит времени превышен — `blocked`.

## Настройки ноды

У `MoveToBlackboardPosition` появились русские поля:

```text
Считать маршрут заблокированным через, секунд
Минимальный заметный прогресс, клеток
Отменять, если цель исчезла
```

Безопасные значения по умолчанию:

```text
stuckTimeoutSeconds = 2.5
minimumProgressCells = 0.05
abortOnTargetLost = true
```

Новая нода сохраняет эти значения сразу. Старый граф без новых полей также получает безопасные значения через человеческую панель редактора.

### `stuckTimeoutSeconds`

Сколько активного времени может пройти без заметного приближения к цели.

```text
0 — отключить автоматическое признание маршрута заблокированным
```

### `minimumProgressCells`

Насколько должно уменьшиться расстояние, чтобы движение считалось настоящим прогрессом.

Мелкие колебания меньше этого порога не сбрасывают таймер.

### `abortOnTargetLost`

Если включено, исчезновение выбранного ключа цели немедленно отменяет текущее движение.

Если ключ всё ещё содержит корректную новую позицию, текущая нода не перенаправляется: она продолжает движение к ранее замороженной цели.

## Реактивная отмена

### Новый приказ игрока

```text
Игрок отдаёт приказ
→ route status = player_override
→ bridge немедленно вызывает обычный runtime update
→ runtime возвращает cancelled
→ старый clear_move содержит старый ownerToken
→ приказ игрока остаётся активным
```

Для этого события bridge **не** передаёт искусственный explicit cancel. Существующая логика runtime сама видит пользовательский приказ и формирует правильный результат.

### Собственный приказ исчез

```text
AI MoveOrder пропал до прибытия
→ route status = order_missing
→ bridge немедленно вызывает обычный runtime update
→ runtime возвращает failure
```

### Цель исчезла

```text
targetKey больше не содержит позиции
→ route status = target_lost
→ bridge передаёт explicit cancel с понятной причиной
→ runtime выполняет lifecycle cancel
→ clear_move удаляет только приказ с совпадающим ownerToken
```

### Нет прогресса

```text
расстояние не уменьшается достаточно долго
→ route status = blocked
→ bridge передаёт explicit cancel
→ runtime отменяет длительное действие
→ token-защищённая очистка снимает только собственный приказ ИИ
```

## Приоритет проверки

```text
пауза
→ прибытие
→ приказ игрока
→ исчезновение собственного приказа
→ исчезновение цели
→ заметный прогресс
→ временная остановка
→ блокировка
```

Прибытие проверяется до исчезновения приказа, чтобы достигшая цели нода не считалась ошибочной только потому, что обычная симуляция уже сняла завершённый приказ.

## Правильная работа паузы

Игра хранит состояние паузы через:

```text
AiTestLabRuntime.getAiTestPaused(state)
```

Route tracker использует именно этот источник, а также режим редактора.

На паузе временные отметки маршрута сдвигаются вперёд. Поэтому время, проведённое на паузе, не засчитывается после продолжения симуляции.

Проверенный сценарий:

```text
начать движение
→ поставить симуляцию на паузу на 5 секунд
→ продолжить
→ через 0.1 секунды активного времени статус ещё stalled
→ ложной блокировки нет
```

## Blackboard

В память выбранного бойца записываются:

```text
active_move_route_status
active_move_no_progress_ms
active_move_last_distance
active_move_abort_code
active_move_abort_reason
```

Эти значения могут использовать будущие универсальные условия и Utility AI score-модификаторы.

Они не раскрывают бойцу объективную информацию о мире. Это состояние собственного действия бойца.

## `След ИИ`

В редакторе отображаются дополнительные строки:

```text
Маршрут: Движение
Без прогресса: 0.4 сек.
```

Для отмены:

```text
Маршрут: Заблокирован
Без прогресса: 2.8 сек.
Причина прерывания: Маршрут заблокирован: боец не продвигается 2,8 сек.
```

Статусы переводятся на русский:

```text
moving          → Движение
stalled         → Нет прогресса
blocked         → Заблокирован
arrived         → Цель достигнута
player_override → Новый приказ игрока
target_lost     → Цель потеряна
order_missing   → Приказ движения исчез
cancelled       → Отменён
idle            → Нет маршрута
```

Debug payload обновляется только для соответствующего `unitId`, поэтому маршрут одного бойца не должен попасть в карточку другого.

## Производительность

Тяжёлый граф ИИ не запускается каждые 60 мс.

Каждый poll выполняет только лёгкие операции:

```text
прочитать активный token
измерить одно расстояние
сравнить timestamps
обновить несколько memory-полей
```

Полный runtime принудительно запускается только при значимом событии:

```text
player_override
order_missing
target_lost
blocked
```

Во всех остальных случаях сохраняется обычный интервал расчёта графа.

## Безопасность приказов

Route tracker никогда не удаляет `MoveOrder` напрямую.

Он только просит runtime выполнить пересчёт или отмену. Фактическая очистка по-прежнему проходит через:

```text
clear_move(ownerToken)
```

Bridge снимает приказ только при совпадении токена.

## Ограничения

Пока отсутствуют:

- карта проходимости;
- столкновение движения со зданиями;
- A* или другой pathfinder;
- waypoint-маршрут;
- перестройка пути;
- определение конкретного препятствия;
- резервирование укрытия;
- реактивная перепроверка всех условий ветки;
- отдельные каналы движения/оружия/позы;
- сохранение route state в сцену;
- массовый scheduler для армии.
