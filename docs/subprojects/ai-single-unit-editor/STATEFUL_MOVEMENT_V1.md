# Stateful AI Movement v1

Дата: 2026-07-12  
Статус: реализовано и проверено  
Подпроект: `ai-single-unit-editor`

## Что появилось

Добавлена первая настоящая action-нода, которая выполняется несколько тиков ИИ:

```text
MoveToBlackboardPosition
Двигаться к позиции из памяти
```

До этого длительной нодой была только `Wait`: она могла ждать, но не управляла настоящим игровым действием. Теперь граф умеет запустить движение, помнить его, продолжать между тиками, завершить по прибытии или безопасно отменить.

## Первый рабочий сценарий

```text
Под огнём
→ найти укрытие
→ пригнуться
→ двигаться к позиции best_cover_position
→ достигнуть позиции
→ лечь
```

`SequenceWithMemory` не начинает эту цепочку заново каждый тик. Пока боец идёт, активным шагом остаётся `MoveToBlackboardPosition`.

## Разделение ответственности

### `AiGraphRuntime`

Чистое ядро ИИ:

- выбирает и замораживает цель;
- создаёт token действия;
- возвращает `running`;
- измеряет расстояние до цели;
- определяет завершение, тайм-аут, провал и отмену;
- выдаёт смысловые effects `begin_move` и `clear_move`.

Он не импортирует:

- DOM;
- PixiJS;
- localStorage;
- `SimulationState`.

### `SimulationTick`

Как и раньше, только симуляция реально меняет координаты бойца. Runtime не дублирует физику движения и не двигает модель сам.

### `AiStatefulMoveGameBridge`

Мост между чистым runtime и игрой:

- создаёт обычный `MoveOrder`;
- помечает его как AI-приказ;
- записывает token владельца;
- публикует текущий приказ в Blackboard;
- очищает приказ только при совпадении token;
- добавляет сведения о цели и остатке пути в `След ИИ`.

## Замороженная цель

При старте нода один раз читает позицию из Blackboard, например:

```text
best_cover_position
```

После этого позиция сохраняется в execution state. Если поиск укрытия на следующем тике выдаст другую точку, уже начатое действие не начнёт дёргать бойца между целями.

Новая цель будет выбрана только после завершения или отмены текущего действия и нового расчёта графа.

## Состояние выполнения

Для движения сохраняются:

```text
targetKey
frozen target {x, y}
acceptanceRadiusCells
timeoutMs
actionToken
start time
last update time
active sequence child
```

Состояние serializable и хранится отдельно от Blackboard.

## Жизненный цикл движения

### `start`

1. Проверяется `self_position`.
2. Читается позиция из выбранного ключа Blackboard.
3. Цель замораживается.
4. Создаётся `actionToken`.
5. Один раз выдаётся effect `begin_move`.
6. Нода возвращает `running`.

### `update`

На следующих тиках:

- новый приказ движения не создаётся;
- читается текущая позиция бойца;
- считается оставшееся расстояние;
- проверяется, существует ли тот же AI-приказ;
- нода продолжает возвращать `running`.

### `complete`

Когда расстояние меньше радиуса достижения:

- выдаётся `clear_move` с тем же token;
- нода получает `complete`;
- `SequenceWithMemory` в этот же расчёт может перейти к следующему шагу, например `Лечь`.

### `cancel`

При явной отмене:

- записывается событие `cancel`;
- execution state очищается;
- выдаётся token-защищённый `clear_move`;
- старый AI-приказ снимается только если он всё ещё принадлежит этой ноде.

## Защита приказа игрока

В `MoveOrder` появились необязательные поля:

```text
source: player | ai
ownerToken
```

Главное правило:

> Устаревшая отмена ИИ никогда не удаляет новый приказ игрока.

Если игрок во время AI-движения отдаёт новый приказ:

1. Blackboard показывает `active_move_source = player`.
2. Runtime завершает старую AI-ноду состоянием `cancelled`.
3. Runtime выдаёт очистку только со старым AI-token.
4. Мост сравнивает token.
5. Новый приказ игрока не совпадает и остаётся активным.

Это отдельно проверено автоматическим bridge-smoke тестом.

## Другие исходы

### Цель отсутствует

Если выбранный ключ Blackboard не содержит конечную позицию `{x, y}`, нода завершается `failure` до создания приказа.

### Цель уже достигнута

Если боец уже находится внутри радиуса достижения, нода сразу завершается и последовательность переходит дальше.

### Приказ ИИ исчез

Если до прибытия собственный приказ движения исчез без приказа игрока, это считается `failure`.

### Тайм-аут

После `timeoutSeconds` нода:

- завершается `failure`;
- выдаёт token-защищённую очистку;
- пишет русскую причину в trace.

### Граф изменён во время движения

Устаревшее execution state отклоняется. Если в нём было движение, runtime также выдаёт безопасную очистку прежнего AI-token.

## Настройки в редакторе

Русский интерфейс по умолчанию показывает:

```text
Цель из памяти
Радиус достижения, клеток
Максимальное время, секунд
```

Готовые цели:

```text
best_cover_position   — лучшая точка укрытия
order_target_position — точка приказа
retreat_position      — точка отхода
```

JSON вручную редактировать не требуется.

Осталось: 7,4 клетки

На графе активная нода подсвечивается жёлтым цветом `running`.

## Ограничения v1

Пока нет:

- поиска маршрута вокруг препятствий;
- честного состояния «путь заблокирован»;
- резервирования укрытия;
- привязки к конкретному объекту укрытия вместо позиции;
- реактивной перепроверки всех условий ветки во время движения;
- параллельных каналов движения, оружия и позы;
- сохранения execution state в JSON сцены;
- автоматического исполнения графа для всей армии.

Движение использует существующую прямолинейную механику `SimulationTick`.

Состоянийная нода должна находиться непосредственным ребёнком `SequenceWithMemory`. Вложенные состоянийные последовательности и несколько независимых длительных цепочек пока отложены.
