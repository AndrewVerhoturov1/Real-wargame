# Stateful AI Movement v1

Дата: 2026-07-12  
Статус: реализовано и проверено  
Подпроект: `ai-single-unit-editor`

## Результат

Добавлена первая настоящая action-нода, которая выполняется несколько тиков ИИ:

```text
MoveToBlackboardPosition
Двигаться к позиции из памяти
```

До этого длительной нодой была только `Wait`: она могла ждать, но не управляла настоящим игровым действием. Теперь граф умеет:

```text
выбрать позицию
→ начать движение один раз
→ помнить действие между тиками
→ показывать оставшееся расстояние
→ завершиться по прибытии
→ перейти к следующему шагу
→ безопасно отмениться
```

## Первый рабочий сценарий

```text
Под огнём
→ найти укрытие
→ пригнуться
→ двигаться к best_cover_position
→ достигнуть позиции
→ лечь
```

`SequenceWithMemory` не начинает цепочку заново каждый тик. Пока боец идёт, активным шагом остаётся `MoveToBlackboardPosition`.

## Разделение ответственности

### `AiGraphRuntime`

Чистое ядро ИИ:

- один раз читает и замораживает цель;
- создаёт token действия;
- возвращает `running`;
- измеряет расстояние до цели;
- определяет завершение, тайм-аут, провал и отмену;
- выдаёт runtime-effects `begin_move` и `clear_move`.

Runtime не импортирует DOM, PixiJS, localStorage или `SimulationState`.

Длительные effects намеренно принадлежат `AiGraphRuntime`, а не мгновенному `AiGraphRunner`. На границе с игрой они проходят проверяющий декодер `readAiGraphRuntimeMoveEffect()`.

### `SimulationTick`

Только симуляция реально меняет координаты бойца. Runtime не дублирует физику движения.

### `AiStatefulMoveGameBridge`

Мост между чистым runtime и игрой:

- создаёт обычный `MoveOrder`;
- помечает AI-приказ token владельца;
- публикует текущий приказ в Blackboard;
- очищает только приказ с совпадающим token;
- не удаляет новый приказ игрока;
- не скрывает следующий шаг последовательности, например `Лечь`;
- добавляет цель и остаток пути в `След ИИ`.

## Замороженная цель

При старте нода один раз читает позицию из Blackboard, например:

```text
best_cover_position
```

После этого позиция сохраняется в execution state. Если поиск укрытия на следующем тике выдаст другую точку, уже начатое действие не дёргает бойца между целями.

Новая цель выбирается только после завершения или отмены текущего действия.

## Состояние выполнения

Для движения сохраняются:

```text
targetKey
frozen target {x, y}
acceptanceRadiusCells
timeoutMs
actionToken
start time
last update time
active sequence child
```

Состояние serializable и хранится отдельно от Blackboard.

## Жизненный цикл

### `start`

1. Проверяется `self_position`.
2. Читается позиция из выбранного ключа Blackboard.
3. Цель замораживается.
4. Создаётся `actionToken`.
5. Один раз выдаётся `begin_move`.
6. Нода возвращает `running`.

### `update`

На следующих тиках:

- новый приказ не создаётся;
- читается текущая позиция;
- считается остаток пути;
- проверяется владелец текущего приказа;
- нода продолжает возвращать `running`.

### `complete`

Когда расстояние не больше радиуса достижения:

- выдаётся token-защищённый `clear_move`;
- записывается `complete`;
- `SequenceWithMemory` в тот же расчёт переходит к следующему шагу;
- очистка движения не перезаписывает состояние следующего действия.

### `cancel`

При отмене:

- записывается lifecycle `cancel`;
- execution state очищается;
- выдаётся `clear_move` со старым token;
- bridge удаляет приказ только при совпадении token.

## Защита приказа игрока

В `MoveOrder` появились необязательные поля:

```text
source: player | ai
ownerToken
```

Главное правило:

> Устаревшая отмена ИИ никогда не удаляет новый приказ игрока.

Реальный правый клик игры создаёт старый совместимый приказ без token. Bridge распознаёт такой приказ как пользовательский. Это отдельно проверено автоматическим тестом.

Если игрок заменяет AI-движение:

1. Blackboard показывает источник `player`.
2. Runtime отменяет старую AI-ноду.
3. Runtime выдаёт очистку только со старым AI-token.
4. Bridge сравнивает token.
5. Новый приказ игрока остаётся активным.

## Другие исходы

### Цель отсутствует

Если выбранный ключ не содержит конечную позицию `{x, y}`, нода возвращает `failure` до создания приказа.

### Цель уже достигнута

Если боец уже внутри радиуса достижения, нода завершается сразу и последовательность идёт дальше.

### Приказ ИИ исчез

Если собственный приказ исчез до прибытия и не был заменён приказом игрока, это `failure`.

### Тайм-аут

После `timeoutSeconds` нода:

- возвращает `failure`;
- выдаёт безопасный `clear_move`;
- записывает русскую причину в trace.

### Граф изменён

Устаревшее execution state отклоняется. Если оно содержало движение, runtime выдаёт очистку прежнего AI-token.

## Настройки в редакторе

Русский интерфейс по умолчанию показывает:

```text
Цель из памяти
Радиус достижения, клеток
Максимальное время, секунд
```

Готовые цели:

```text
best_cover_position    — лучшая точка укрытия
order_target_position — точка приказа
retreat_position      — точка отхода
```

Новая нода из палитры немедленно получает и сохраняет безопасные значения:

```text
targetKey = best_cover_position
acceptanceRadiusCells = 0.2
timeoutSeconds = 15
```

Редактировать JSON и отдельно сохранять значения по умолчанию не требуется.

## Живая диагностика

Для активной ноды отображаются:

```text
Состояние: Выполняется
Активная нода: Двигаться к укрытию
Выполняется: 3,2 сек.
Цель из памяти: best_cover_position
Координаты цели: 18,5; 12,5
Осталось: 7,4 клетки
```

Активная нода подсвечивается жёлтым цветом `running`.

## Ограничения v1

Пока нет:

- pathfinding вокруг препятствий;
- честного состояния «путь заблокирован»;
- резервирования укрытия;
- привязки к конкретному объекту укрытия;
- полной реактивной перепроверки ветки;
- параллельных каналов движения, оружия и позы;
- сохранения execution state в JSON сцены;
- автоматического исполнения графа для всей армии.

Движение использует существующую прямолинейную механику `SimulationTick`.

Состоянийная нода должна быть непосредственным ребёнком `SequenceWithMemory`. Вложенные состояния и несколько независимых длительных цепочек отложены.
