# CI Audit — 2026-07-18

## Executive finding

The repository had path-aware workflows in several places, but the effective process was still broad and SHA-centric. `Preview Core Checks` ran on every preview PR without path filters, while multiple subsystem workflows repeated their own `npm ci`, overlapping smoke suites and production build. Performance workflows were automatically attached to broad runtime paths. Agent instructions and the result template then treated exact-head performance evidence as a default completion requirement.

The primary waste was therefore cumulative: broad triggers + duplicated setup/build + agent-side exact-head rules.

## Non-workflow bureaucracy found

- `.agents/skills/real-wargame-performance/SKILL.md` instructed agents to run an exact-head enforced browser performance workflow whenever an interactive hot path was touched, without first asking whether the final change altered the measured scenario.
- `docs/orchestration/RESULT_TEMPLATE.md` had an `exact-head enforced workflow` field, encouraging executors to treat exact-head repetition as a mandatory proof item.
- PR reports repeatedly used “exact-head matrix” as the acceptance boundary even after documentation or transport-only commits.
- Temporary evidence-publishing workflows and cleanup commits produced new SHAs that invalidated evidence under the old convention.

## Implemented simplification

- One automatic `PR Risk CI` classifies paths and runs one minimal matrix.
- Automatic runtime verification performs at most one `npm ci` and one build.
- Documentation-only changes run documentation/policy checks without build or Chromium.
- Previous broad workflows remain manually available rather than being deleted.
- Browser performance is label/manual only and rejects a missing or generic reason.
- A tested implementation SHA may remain valid through a machine-checked, narrow documentation tail.
- PR and result templates now require an explicit distinction between required, risk-selected and deliberately omitted checks.

## Expected typical reduction

| Typical PR | Before | After |
|---|---|---|
| Documentation-only | Preview Core install + large smoke/build matrix, plus docs checks | No game build, no browser/performance; documentation/policy only |
| Isolated navigation change | Preview Core + Command Route + Navigation + often Directional/Combat and repeated builds | One install, TypeScript, four focused navigation checks, one build |
| Tactical UI change | Preview Core + Compact Route + Tactical Order + overlapping route/runtime/build | One install, focused tactical/UI checks, one build |
| Runtime hot-path change | Several core workflows plus both heavy performance workflows | Focused automatic matrix; only the explicitly justified performance scenario is opt-in |

For the common cases, automatic `npm ci` and build count drops from several independent jobs to at most one. Chromium and long performance runs drop from automatic broad-path execution to zero unless explicitly justified.
