# Идеи развития нодовой системы поведения ИИ

Дата: 2026-07-10  
Статус: исследовательский документ / источник идей для будущей архитектуры  
Подпроект: `ai-single-unit-editor` / Soldier Tactical Awareness Lab

## Цель документа

Собрать полезные возможности трёх сильных систем визуального построения и исполнения поведения ИИ, сопоставить их с текущим состоянием Real-Wargame и определить, какие идеи стоит внедрять.

Рассмотренные системы:

1. **Unreal Engine AI Stack** — StateTree, Behavior Tree, Environment Query System, AI Perception и Smart Objects.
2. **Unity Behavior** — современный визуальный редактор поведения, длительные действия, события, подграфы и сохранение исполнения.
3. **BehaviorTree.CPP + Groot2** — строгая модель исполнения Behavior Tree и мощная запись, отладка и воспроизведение поведения.

Главный принцип: **не заменять существующий UtilitySelector обычным Behavior Tree**. Наиболее сильная архитектура для Real-Wargame — гибрид:

```text
StateTree-подобные состояния для общего режима бойца
+
Utility AI для выбора лучшего варианта
+
Behavior Tree / подграф для пошагового исполнения
+
EQS-подобные запросы для поиска позиций и объектов
+
Groot-подобная история, запись и отладка
```

# 2. Фундаментальные возможности исполнителя

## 1. Состояние ноды «Выполняется»

### Суть

Действие может занимать время и возвращать не только успех или провал, но и состояние `Running` — «всё ещё выполняется».

Примеры:

- движение к укрытию;
- ползание;
- перезарядка;
- прицеливание;
- наблюдение сектора;
- подавляющий огонь;
- ожидание команды.

### Что даст

Граф сможет реально выполнять последовательности во времени:

```text
Найти укрытие
→ начать движение
→ двигаться несколько секунд
→ прибыть
→ лечь
→ открыть огонь
```

### Приоритет

**Критический.** Следующий фундаментальный этап GraphRunner.

## 2. Состояние «Ожидает»

### Суть

Отдельное состояние для ноды, которая не выполняет активную работу, а ждёт:

- события;
- разрешения;
- окончания задержки;
- освобождения ресурса;
- сообщения командира.

### Что даст

Позволит не опрашивать всё каждый кадр и ясно отличать активную работу от ожидания.

### Приоритет

**Высокий.** Делать вместе с событийной системой.

## 3. Полный жизненный цикл действия

### Суть

Каждая длительная нода должна иметь этапы:

```text
setup       — подготовка постоянных данных
start       — запуск действия
update      — продолжение
end         — нормальное завершение или прерывание
teardown    — очистка ресурсов
serialize   — сохранение внутреннего состояния
deserialize — восстановление
```

### Что даст

Нода «Двигаться к укрытию» сможет:

- один раз построить маршрут;
- начать перемещение;
- проверять прибытие;
- отменить маршрут;
- освободить зарезервированную точку;
- сохранить прогресс.

### Приоритет

**Критический.** Делать вместе с `Running`.

## 4. Контракт отмены действия

### Суть

При смене решения старая нода должна не просто исчезать, а корректно завершаться через `cancel / abort / halt`.

### Что даст

- не останутся висящие маршруты;
- не будут одновременно действовать несовместимые команды;
- зарезервированное укрытие освободится;
- анимация, поза, цель и состояние оружия не разойдутся с графом.

### Приоритет

**Критический.** Без отмены длительные действия опасны.

## 5. Реактивные условия

### Суть

Условия, предшествующие длительному действию, продолжают перепроверяться во время его исполнения.

```text
Пока враг виден
и оружие готово
и огонь разрешён
→ вести огонь
```

Если враг исчез или огонь запрещён, действие немедленно отменяется.

### Что даст

Быструю реакцию на изменение боя без ожидания завершения старого действия.

### Приоритет

**Критический для тактического поведения.**

## 6. Наблюдающие условия и прерывание ветви

### Суть

Условие подписывается на изменение конкретного параметра Blackboard и будит граф только при значимом изменении.

### Что даст

- меньше постоянных полных перерасчётов;
- немедленная реакция на угрозу;
- чистый журнал событий;
- лучшая масштабируемость.

### Приоритет

**Очень высокий.**

## 7. Событийное исполнение

### Суть

Граф запускается не только по таймеру, но и по событиям:

- враг замечен;
- враг потерян;
- получен выстрел;
- выросло подавление;
- закончились патроны;
- получен новый приказ;
- маршрут заблокирован;
- укрытие разрушено;
- командир погиб.

### Что даст

Меньше лишней работы и более естественное поведение.

### Приоритет

**Очень высокий перед масштабированием на много бойцов.**

## 8. Иерархические состояния

### Суть

Поверх Utility AI вводится уровень общего режима бойца:

```text
Нормальное состояние
├─ выполнение приказа
├─ наблюдение
└─ отдых

Контакт
├─ поиск позиции
├─ огонь
└─ смена укрытия

Критическое состояние
├─ подавлен
├─ ранен
├─ паника
└─ отход
```

### Что даст

UtilitySelector сравнивает только допустимые для текущего режима действия. Раненый или паникующий боец не будет оценивать обычный штурм наравне с самосохранением.

### Приоритет

**Очень высокий после `Running`.**

## 9. Явные переходы между состояниями

### Суть

Переходы задаются отдельно и могут происходить:

- при успехе;
- при провале;
- при завершении;
- по событию;
- по условию;
- по таймауту.

### Что даст

```text
Иду к укрытию
├─ достиг → Занимаю укрытие
├─ путь потерян → Ищу новый путь
├─ укрытие занято → Выбираю другое
├─ ранен → Реагирую на ранение
└─ враг подошёл близко → Ближний бой
```

### Приоритет

**Высокий.**

## 10. Разделение выбора и исполнения

### Суть

```text
Общее состояние
→ ограничивает допустимое поведение

UtilitySelector
→ выбирает лучший вариант

Подграф
→ исполняет выбранный вариант по шагам

Action Runtime
→ реально двигает, целится, стреляет и ждёт
```

### Что даст

Чистую архитектуру без попытки заставить UtilitySelector одновременно выбирать и выполнять многошаговые действия.

### Приоритет

**Обязательный архитектурный принцип.**

# 3. Структура больших графов

## 11. Подграфы поведения

### Суть

Один граф можно использовать как ноду другого графа.

Примеры библиотечных подграфов:

- занять укрытие;
- перезарядиться;
- отреагировать на обстрел;
- проверить сектор;
- подавлять позицию;
- следовать за командиром;
- отходить перебежками.

### Что даст

- небольшие и читаемые графы;
- переиспользование поведения;
- единое исправление общей процедуры;
- библиотеку поведения для разных ролей.

### Приоритет

**Очень высокий.**

## 12. Статические подграфы

### Суть

Конкретный подграф задан заранее в основном графе.

### Что даст

Простой первый этап переиспользования поведения.

### Приоритет

**Высокий. Делать раньше динамических.**

## 13. Динамические подграфы

### Суть

Подграф выбирается во время игры через параметр или роль бойца.

### Что даст

Одна общая основа сможет подключать разные пакеты поведения:

- стрелок;
- пулемётчик;
- снайпер;
- сапёр;
- командир;
- санитар.

### Приоритет

**Средне-высокий.**

## 14. Ноды-модификаторы

### Суть

Модификатор оборачивает дочернюю ветвь:

- повторить;
- повторять до успеха;
- повторять до провала;
- инвертировать результат;
- считать успехом;
- выполнить один раз;
- задержать запуск;
- ограничить таймаутом.

### Что даст

Меньше специальных нод и более выразительный граф.

### Приоритет

**Высокий.**

## 15. Таймаут действия

### Суть

Длительная нода автоматически проваливается или отменяется, если выполняется слишком долго.

### Что даст

Боец не застрянет навсегда при:

- недостижимой точке;
- потерянном маршруте;
- разрушенном укрытии;
- исчезнувшей цели;
- невозможном приказе.

### Приоритет

**Высокий и сравнительно дешёвый после `Running`.**

## 16. Повтор после временного провала

### Суть

Дочернее действие повторяется ограниченное число раз с задержкой.

### Что даст

- повторный поиск пути;
- выбор другого места в укрытии;
- повторная передача сообщения;
- повторный поиск цели.

### Приоритет

**Средний.**

## 17. Последовательность с памятью

### Суть

Последовательность помнит уже выполненные шаги и после временного провала продолжает с проблемного места.

### Что даст

Не придётся повторно выбирать и резервировать уже выбранную позицию после кратковременного сбоя маршрута.

### Приоритет

**Средне-высокий.**

## 18. Реактивная последовательность

### Суть

Предыдущие условия продолжают проверяться, пока выполняется последующее действие.

### Что даст

Немедленную отмену огня, движения или ожидания при изменении условий.

### Приоритет

**Высокий.**

## 19. Асинхронная последовательность с точками уступки

### Суть

После каждого шага исполнитель отдаёт управление другим системам и только затем переходит к следующему.

### Что даст

Позволит между «прицелиться» и «выстрелить» обработать исчезновение цели, ранение или новый приказ.

### Приоритет

**Средний.**

## 20. Переключатель по значению

### Суть

Нода `Switch` выбирает ветвь по значению параметра.

Применение:

- тип приказа;
- текущая поза;
- роль бойца;
- состояние оружия;
- уровень ранения.

### Что даст

Компактные графы вместо длинных цепочек Selector и условий.

### Приоритет

**Средний.**

## 21. Нода «Если / иначе»

### Суть

Явное ветвление по одному условию.

```text
Если оружие заряжено
→ стрелять
Иначе
→ перезарядиться
```

### Что даст

Повысит читаемость простых развилок.

### Приоритет

**Средний.**

# 4. Параллельное поведение

## 22. Простое параллельное поведение

### Суть

Одна главная задача и одна фоновая ветвь.

```text
Пока двигаюсь к позиции
→ наблюдать за угрозой
```

### Что даст

Большую часть полезного параллелизма без сложности произвольного запуска многих ветвей.

### Приоритет

**Высокий. Делать раньше полного Parallel.**

## 23. Полное параллельное выполнение

### Суть

Несколько нод одновременно находятся в состоянии `Running`.

### Что даст

Боец сможет одновременно:

- двигаться;
- смотреть на угрозу;
- поддерживать строй;
- говорить;
- следить за подавлением.

### Приоритет

**Высокий, но только после отмены и владения каналами.**

## 24. Владение каналами действия

### Суть

Действия занимают отдельные каналы:

- перемещение;
- поза;
- поворот корпуса;
- наведение оружия;
- огонь;
- речь;
- взаимодействие.

Две ноды могут выполняться вместе, только если не спорят за один канал.

### Что даст

Можно одновременно идти, целиться и говорить, но нельзя идти в две точки или одновременно стоять и лежать.

### Приоритет

**Обязателен перед полным параллелизмом.**

## 25. Условия завершения Parallel

### Суть

Настраивается:

- сколько успехов нужно;
- сколько провалов достаточно;
- ждать ли всех;
- отменять ли оставшиеся ветви.

### Что даст

Гибкое объединение нескольких действий и проверок.

### Приоритет

**Средний.**

## 26. Ноды объединения ветвей

### Суть

- ждать завершения всех;
- ждать завершения любой;
- продолжить после первого успеха;
- продолжить после первого провала.

### Что даст

Можно дождаться готовности оружия, прибытия в точку и разрешения командира перед открытием огня.

### Приоритет

**Позже, после безопасного параллелизма.**

# 5. Тактические запросы наподобие EQS

## 27. Конвейер «кандидаты → фильтры → оценки»

### Суть

Тактический запрос разделяется на этапы:

```text
Генератор
→ создаёт кандидатов

Жёсткие фильтры
→ удаляют невозможные

Оценки
→ начисляют очки оставшимся

Контекст
→ задаёт, относительно кого считать
```

### Что даст

Настоящий визуальный конструктор выбора позиции, а не одна непрозрачная команда `FindBestObject`.

### Приоритет

**Критически важный для тактической части.**

## 28. Несколько генераторов кандидатов

### Суть

Кандидаты создаются разными способами:

- сетка вокруг бойца;
- точки вдоль маршрута;
- места около укрытий;
- край леса;
- обратный склон;
- фланги цели;
- точки вокруг приказа;
- готовые слоты предметов.

### Что даст

Поиск не только существующих объектов, но и подходящего места в пространстве.

### Приоритет

**Высокий.**

## 29. Контексты запроса

### Суть

Одна проверка может считаться относительно:

- бойца;
- текущего врага;
- всех известных врагов;
- командира;
- цели приказа;
- союзников;
- выбранного маршрута.

### Что даст

Переиспользуемые ноды расстояния, видимости, укрытия и направления.

### Приоритет

**Высокий.**

## 30. Жёсткие фильтры

### Суть

Полностью исключают невозможный вариант:

- путь отсутствует;
- позиция занята;
- слот не подходит;
- точка за пределами приказа;
- позиция физически недоступна.

### Что даст

Utility AI не выберет невозможное действие из-за высоких очков по другим факторам.

### Приоритет

**Высокий.**

## 31. Мягкие оценки

### Суть

Не запрещают вариант, а меняют его привлекательность:

- расстояние;
- маскировка;
- качество обзора;
- опасность маршрута;
- высота;
- сектор огня.

### Что даст

Гибкий и объяснимый выбор лучшего среди допустимых вариантов.

### Приоритет

**Высокий.**

## 32. Просмотр всех кандидатов на карте

### Суть

Каждый кандидат показывается с итоговым результатом и причинами:

```text
Точка A: 82
+35 защита
+24 короткий маршрут
+18 сектор огня
+5 маскировка

Точка B: исключена
Причина: занята
```

### Что даст

Пользователь увидит, почему выбрана именно эта позиция.

### Приоритет

**Очень высокий для лаборатории.**

## 33. Пошаговый просмотр тестов запроса

### Суть

Показывать кандидатов после каждого этапа:

1. все созданные точки;
2. после проверки пути;
3. после проверки видимости;
4. после проверки укрытия;
5. после оценки опасности;
6. итог.

### Что даст

Легко найти тест, который ошибочно удалил все варианты.

### Приоритет

**Высокий.**

## 34. Именованные параметры запроса

### Суть

Один запрос получает настройки:

- радиус поиска;
- максимум длины маршрута;
- минимальную защиту;
- допустимую опасность;
- вес высоты;
- вес маскировки.

### Что даст

Один запрос будет работать по-разному для смелого, осторожного, раненого или подавленного бойца.

### Приоритет

**Средне-высокий.**

## 35. Ограничение стоимости запроса

### Суть

У запроса задаются:

- максимум кандидатов;
- максимум дорогих проверок;
- лимит миллисекунд;
- частота обновления;
- кэш по revision;
- ранний выход при достаточно хорошем результате.

### Что даст

Предсказуемую производительность и возможность масштабирования.

### Приоритет

**Высокий до массового ИИ.**

## 36. Уровни качества тактического поиска

### Суть

Разные режимы расчёта:

- быстрый;
- обычный;
- точный;
- лабораторный полный.

### Что даст

- число отмен;
- долю бездействия;
- число дорогих запросов;
- частоту зацикливания;
- выполнение приказа;
- выживание.

### Что даст

Объективное сравнение двух версий графа.

### Приоритет

**Средне-высокий.**

# 14. Самые важные пробелы текущей системы

В первую очередь полезно реализовать:

1. `Running` и `Waiting`.

4. Реактивные условия и прерывание ветви.

8. Типизированные входы, выходы и контракты параметров.
9. Историю решений и изменений Blackboard.
10. Временную шкалу, запись и воспроизведение.
11. Точки останова и принудительный результат условий.
12. Конвейер тактического запроса: кандидаты → фильтры → оценки.
13. Визуальный разбор всех кандидатов позиции.
14. Точные слоты, вместимость и резервирование укрытий.

# 17. Что не следует делать

- Не заменять Utility AI чистым Behavior Tree.
- Не запускать полный граф каждого бойца каждый кадр без событий и бюджета.
- Не добавлять полный параллелизм до появления отмены и владения каналами.
- Не вводить несколько бойцов до стабилизации одиночного исполнителя.
- Не скрывать тактический поиск внутри одной непрозрачной ноды.
- Не смешивать выбор решения, длительное исполнение и PixiJS-отрисовку.
- Не хранить всю временную информацию в одном плоском Blackboard.
- Не менять формат графа без миграции старых данных.

# Итог

Real-Wargame уже имеет сильную основу выбора решений: субъективные знания бойца, awareness, UtilitySelector, veto, инерцию, оценки и объяснение результата.

Следующий качественный скачок даст не увеличение количества специальных нод, а превращение `AiGraphRunner` в **состоянийный, событийный и записываемый исполнитель длительного поведения**.

Целевая формула:

```text
StateTree для общего режима
+
Utility AI для выбора
+
Behavior Tree / подграфы для исполнения
+
EQS-подобные запросы для тактического пространства
+
Groot-подобная запись и отладка
```
